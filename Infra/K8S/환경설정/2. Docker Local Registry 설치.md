# Docker Local Registry 설치

Kubernetes에서 Private Registry를 운영할 때는 Registry Container 하나를 띄우는 것보다 **영속 Storage, TLS, 인증, 모든 Node의 Runtime 신뢰 설정, Backup과 Garbage Collection**을 함께 설계해야 한다. Docker CLI가 Push에 성공해도 Kubelet이 사용하는 containerd가 Pull할 수 있다는 뜻은 아니다.

## Docker Private Registry 설치
```shell
docker volume create registry-data
docker pull registry:3
docker run --restart always --detach \
  --publish 5000:5000 \
  --mount type=volume,source=registry-data,target=/var/lib/registry \
  --name local-registry \
  registry:3
docker ps
```

Version을 고정하고 Registry 데이터를 Container Layer가 아닌 Volume에 둔다. 이 예시는 학습용 최소 구성이다. 운영에서는 TLS, 인증, 외부 Storage 또는 Volume Backup, Resource 제한과 Monitoring이 필요하다.
아래와 같이 출력된다.
```
CONTAINER ID   IMAGE      COMMAND                   CREATED         STATUS         PORTS                                       NAMES
15e523a02349   registry   "/entrypoint.sh /etc…"   4 seconds ago   Up 3 seconds   0.0.0.0:5000->5000/tcp, :::5000->5000/tcp   ieee160921-registry
```
### Docker Image에 Tag 설정
#### Docker Image Pull
```shell title="hello-world image pull 받는다."
docker pull hello-world
```
#### 이미지에 태그 설정
외부에서 리포지토리에 push 하려면 Tag를 달아야한다.
```shell title="docker image에 tag설정 예시"
docker tag hello-world 192.168.0.41:5000/hello-world
```
### HTTP Registry는 개발망에서만 사용

가장 좋은 방식은 내부 CA가 서명한 TLS Certificate를 배포하고 Docker와 containerd가 그 CA를 신뢰하게 하는 것이다. 다음 `insecure-registries` 설정은 통신 상대와 Image Manifest를 TLS로 검증하지 않으므로 격리된 임시 개발 환경에만 사용한다. Docker Daemon 설정 경로는 일반적으로 `/etc/docker/daemon.json`이며 `~/.docker/daemon.json`이 아니다.

```json title="~/.docker/daemon.json 파일에 insecure-registries 추가"
{
  "builder": {
    "gc": {
      "defaultKeepStorage": "20GB",
      "enabled": true
    }
  },
  "experimental": false,
  "insecure-registries": [
    "192.168.0.41:5000"
  ]
}
```
Local Registry **Server(Master와 Worker 모두에서)** 쪽에서도 똑같이 설정 후 서비스 재시작
```shell title="server에서 docker 재시작"
systemctl restart docker
```
### containerd Registry 설정

과거의 `registry.mirrors` 설정은 Deprecated되었다. 현재 방식은 `config_path`와 Registry별 `hosts.toml`을 사용한다. containerd 1.x와 2.x는 Plugin 경로가 다르므로 먼저 Version과 현재 설정을 확인한다.

```shell
containerd --version
containerd config dump | less
```

containerd 1.x의 예시는 다음과 같다.

```toml title="/etc/containerd/config.toml"
version = 2

[plugins."io.containerd.grpc.v1.cri".registry]
  config_path = "/etc/containerd/certs.d"
```

Registry Host Namespace Directory에 설정한다.

```toml title="/etc/containerd/certs.d/192.168.0.41:5000/hosts.toml"
server = "http://192.168.0.41:5000"

[host."http://192.168.0.41:5000"]
  capabilities = ["pull", "resolve", "push"]
```

TLS Registry라면 `https`를 사용하고 `ca`에 내부 CA 경로를 지정한다. `skip_verify = true`는 Hostname과 Certificate 검증을 끄므로 운영 해법으로 사용하지 않는다.

```shell
systemctl restart containerd
crictl pull 192.168.0.41:5000/hello-world:latest
```

> [!INFO] 위 과정 꼭 진행할것!
> Kubelet이 Image를 Pull하는 모든 Worker Node에서 Registry DNS, Port, CA와 인증을 사용할 수 있어야 한다. `docker pull`이 아니라 `crictl pull`과 실제 Pod 생성으로 검증한다.
### Image Push
```shell title="docker image Push"
docker push 192.168.0.41:5000/hello-world
```
### Local Repository 확인 및 삭제 방법
#### 내부 리포지토리 확인
```shell title="docker local registry 내부 리포지토리 확인"
curl -XGET 192.168.0.41:5000/v2/_catalog
```
#### 안전한 삭제와 공간 회수

`/var/lib/registry` 아래 Directory를 직접 삭제하면 Layer 참조가 깨질 수 있다. Registry API로 Manifest Digest를 삭제한 뒤 읽기 전용 또는 중지 상태에서 Garbage Collection을 수행한다. 실행 전 Backup과 `--dry-run` 결과를 확인한다.

## Kubernetes 인증

인증이 있는 Registry는 같은 Namespace에 `kubernetes.io/dockerconfigjson` Type의 Secret을 만들고 Pod 또는 ServiceAccount의 `imagePullSecrets`에서 참조한다. Secret YAML을 Git에 평문으로 Commit하지 않는다.

## 운영 완료 기준

1. 모든 Worker Node에서 Digest로 Image Pull이 성공한다.
2. Registry를 재시작해도 Image가 유지된다.
3. 인증 없는 Push와 Catalog 조회가 거부된다.
4. TLS Certificate Rotation 절차가 검증되어 있다.
5. Registry Backup을 새 Instance에 복원해 Pull할 수 있다.
6. Disk 사용량, Push/Pull 오류와 Certificate 만료를 Monitoring한다.
