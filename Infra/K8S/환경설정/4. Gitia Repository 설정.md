# Gitea에 Helm Chart Repository 구성하기

Gitea는 Git Repository를 제공하고, Helm Chart는 그 Repository 안에서 일반 Source File로 관리할 수 있다. 다만 Git Repository 자체가 Helm Chart Repository Protocol을 제공하는 것은 아니다. GitOps Controller가 Git에서 Chart를 직접 읽는 방식과, Package Registry에 Chart Package를 발행하는 방식을 구분한다.

## 회원가입
`http://<Gitia IP>:30480/` 경로로 접속
**Register**에서 운영 계정을 만든 뒤 로그인한다. 우측 상단의 `+` 메뉴에서 **New Repository**를 선택하고 Helm Chart를 저장할 Repository 이름과 공개 범위를 지정한다.

Repository 생성 후 표시되는 Clone URL을 사용한다.

```shell
git clone <Git 주소>
cd <Repository 이름>
```

폐쇄망에서는 DNS와 인증서 신뢰 설정을 먼저 확인하고, 개인 계정보다 배포 전용 계정 또는 Deploy Key를 사용한다. Write 권한은 CI에만 주고 Argo CD 같은 배포 주체에는 Read-only 권한을 부여한다.
### 저장소 설정 (우리는 helm으로 배포할 예정이라 아래와 같다.)
```shell title="아래와 같이 배포할 저장소 init"
[root@exscms1 ieee160921]# ls
Chart.yaml  templates  values.yaml
```
### values.yaml 파일 수정
Image Registry 주소를 환경별 Values File에서 설정한다. `latest`는 동일한 Git Revision에서도 실행 Image가 바뀔 수 있으므로 Build Version이나 Git SHA로 고정한다.
```yaml title="values.yaml"
image:
  repository: "10.19.2.171:5000/ieee160921-scms"
  tag: "1.4.2"
```
### templates/ingress.yaml 파일 수정
Ingress Host와 Path를 환경 값으로 분리하고 Template에 주소를 직접 적지 않는다.

```yaml title="values-production.yaml"
ingress:
  enabled: true
  className: nginx
  hosts:
    - host: scms.internal.example
      paths:
        - path: /
          pathType: Prefix
```

## 변경 검증과 배포 흐름

```shell
helm lint .
helm template scms . --values values-production.yaml > rendered.yaml
kubectl apply --dry-run=server --filename rendered.yaml
```

Merge Request에서 Rendering 결과와 Kubernetes Schema를 검증하고, 승인된 Branch나 Tag만 배포 환경이 추적하게 한다. CI가 직접 `helm upgrade`하는 Push 방식과 Argo CD가 Git 상태를 감시하는 Pull 방식이 동시에 같은 Release를 관리하지 않도록 소유권을 하나로 정한다.

## Secret을 Git에 넣지 않기

`values.yaml`에는 Password, Registry Token과 TLS Private Key를 평문으로 Commit하지 않는다. External Secrets, SOPS 같은 암호화 Workflow 또는 폐쇄망 Secret Manager를 사용한다. 과거 Commit에 들어간 Secret은 현재 File에서 지워도 Git History에 남으므로 Credential을 폐기하고 History 정리 여부를 판단해야 한다.

## 운영 점검

- Gitea Repository와 Package 데이터 Backup 및 복원 Test
- Deploy Key·Token 만료와 Rotation
- Branch Protection과 Review 규칙
- Chart Version과 Application Version의 추적 가능성
- Gitea 장애 중 기존 배포의 지속성과 신규 배포 차단 방식
