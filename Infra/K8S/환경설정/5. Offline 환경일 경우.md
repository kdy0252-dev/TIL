# 폐쇄망 Kubernetes 배포 설계

폐쇄망 배포는 `imagePullPolicy` 하나를 바꾸는 작업이 아니다. Cluster가 외부 DNS, Container Registry, Helm Repository, OS Package 저장소, CDN, 인증서 폐기 확인과 시간 동기화에 의존하는 지점을 모두 찾아 내부 공급 경로를 만들어야 한다.

## 먼저 Artifact 목록을 고정한다

- Kubernetes와 Container Runtime Binary·Package
- Application Image와 Init Container Image
- CNI, CSI, Ingress, DNS, Monitoring Image
- Helm Chart와 CRD
- Frontend JavaScript, Font와 Source Map
- OS Package, CA Certificate, License와 취약점 DB

Tag만 기록하지 말고 Digest, Version, 출처와 SHA-256 Checksum을 Manifest로 남긴다. Multi-architecture Image는 필요한 Platform Manifest와 Layer가 모두 반입되었는지 확인한다.

## 내부 Registry를 단일 공급점으로 사용하기

Internet이 연결된 반입 환경에서 Image를 검증하고 내부 Registry로 복사한다. 각 Node에 수동으로 Pre-pull하는 방식은 Node 증설·교체 때 누락되기 쉬우므로 내부 Registry가 기본이고 Pre-pull은 보조 수단이다.

```yaml
image:
  repository: registry.internal.example/order-api
  tag: "1.4.2"
  pullPolicy: IfNotPresent
```

`IfNotPresent`는 Local에 같은 Image Reference가 있으면 Cache를 사용한다. 같은 Tag를 덮어쓰면 Node마다 다른 Image가 실행될 수 있으므로 불변 Tag나 Digest를 사용한다. `Never`는 Image가 없는 Node에서 즉시 실패하므로 완전한 Preload 정책이 있을 때만 선택한다.

## 모든 Node가 같은 조건인지 검증하기

Scheduler는 Image가 있는 Node를 자동으로 골라 주지 않는다. Pod가 어느 Node에 배치되어도 시작할 수 있어야 한다.

```shell
crictl pull registry.internal.example/order-api:1.4.2
crictl images
```

DaemonSet으로 Image를 미리 받는 방식도 결국 Registry 접근과 Runtime별 Image Store를 이해해야 한다. Docker로 받은 Image는 containerd CRI Namespace에서 보이지 않을 수 있으므로 Kubelet이 사용하는 Runtime을 기준으로 확인한다.

## Helm Chart와 Dependency 반입

```shell
helm dependency build ./chart
helm package ./chart
helm template order-api ./order-api-1.4.2.tgz \
  --values values-offline.yaml > rendered.yaml
```

Chart의 하위 Dependency와 Hook가 참조하는 Image까지 목록화한다. Chart 안의 `docker.io`, `quay.io`, `ghcr.io` 문자열을 검색해 내부 주소로 Override되지 않은 Image가 없는지 검사한다.

## Web Asset과 Runtime 외부 호출

CDN의 JavaScript와 Font는 Build Artifact에 포함하거나 내부 Static Server에서 제공한다. 단순 Download만 하지 말고 License, Cache Header, CSP와 SRI 설정을 함께 갱신한다.

Application이 실행 중 외부에 호출하는 다음 항목도 점검한다.

- OAuth/OIDC Discovery와 JWKS Endpoint
- 지도, 결제, Mail, SMS API
- NTP, DNS와 Certificate Chain
- Telemetry·Crash Reporting와 License Server
- Vulnerability Scanner의 DB Update

## 보안 Update와 반입 절차

폐쇄망은 외부 공격 면을 줄이지만 취약점이 자동으로 사라지지는 않는다. 반입 Zone에서 Malware·Signature·SBOM·취약점을 검사하고 승인된 Artifact만 내부 Registry에 Promotion한다. 긴급 CVE를 반입하고 Rollback하는 목표 시간을 운영 절차로 정한다.

## 실제 폐쇄망 Test

외부 Network가 연결된 상태에서 내부 주소만 사용한다고 폐쇄망 검증이 끝난 것은 아니다. Egress를 실제 차단한 Test 환경에서 새 Node를 추가하고 Cluster Add-on과 Application을 처음부터 설치한다. DNS Query, `ImagePullBackOff`, Browser Network 요청과 Application Timeout Log를 관찰해 숨은 의존성을 찾는다.
