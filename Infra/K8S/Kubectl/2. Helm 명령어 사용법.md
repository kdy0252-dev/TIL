# Helm 명령을 배포 흐름으로 이해하기

Helm은 Kubernetes Manifest를 Parameter화한 Chart와, Cluster에 설치된 결과인 Release를 관리한다. Chart Directory 이름과 Release 이름, Kubernetes Deployment 이름은 서로 다른 개념이다.

```text
Chart + values → Rendering된 Manifest → Kubernetes 적용 → Release 이력
```

## 설치 전 검증

Cluster에 바로 적용하기 전에 Chart 문법과 Rendering 결과를 확인한다.

```shell
helm lint ./chart
helm template order-api ./chart \
  --namespace production \
  --values values-production.yaml > rendered.yaml

kubectl apply --dry-run=server --filename rendered.yaml
```

`helm lint`는 Chart 구조를, `helm template`은 값이 합쳐진 최종 YAML을, Server-side Dry Run은 현재 Cluster API와 Admission 정책을 확인한다. Secret 값이 Rendering 결과와 CI Log에 남지 않도록 주의한다.

## 처음 설치하기

```shell
helm install order-api ./chart \
  --namespace production \
  --create-namespace \
  --values values-production.yaml \
  --atomic \
  --wait \
  --timeout 10m
```

첫 번째 인자는 Release 이름, 두 번째 인자는 Chart 경로 또는 Repository Chart 이름이다. 기존 문서처럼 `values.yaml` 경로를 Chart 인자로 전달하면 안 된다. `--atomic`은 설치 실패 시 생성 Resource를 정리하며 자동으로 `--wait`를 사용한다.

## 설치와 Upgrade를 하나의 명령으로 다루기

```shell
helm upgrade --install order-api ./chart \
  --namespace production \
  --create-namespace \
  --values values-production.yaml \
  --atomic \
  --timeout 10m
```

CI/CD에서는 Release 존재 여부와 관계없이 같은 명령을 사용할 수 있다. `--set image.tag=...`는 간단한 Override에는 편하지만 Shell Escape와 Type 변환 문제가 생길 수 있어 복잡한 값은 Version 관리되는 Values File을 사용한다.

## 상태와 실제 값 확인

```shell
helm list --namespace production
helm status order-api --namespace production
helm get values order-api --namespace production --all
helm get manifest order-api --namespace production
helm history order-api --namespace production
```

Repository의 Values와 실제 Release 값이 다를 수 있다. 장애 분석에서는 `helm get values`와 `helm get manifest`로 Cluster에 적용된 결과를 먼저 확인한다.

## Rollback

```shell
helm history order-api --namespace production
helm rollback order-api 3 \
  --namespace production \
  --wait \
  --timeout 10m
```

Rollback은 Kubernetes Manifest를 이전 Revision으로 되돌리지만 Database Migration, 외부 Message 발행과 Persistent Volume 데이터까지 되돌리지 않는다. 이전 Application이 새 Schema와 호환되는지 배포 전에 검증해야 한다.

## 삭제

```shell
helm uninstall order-api --namespace production
```

Release가 만든 Resource를 삭제하지만 `helm.sh/resource-policy: keep`가 붙은 Resource나 Retain 정책의 Persistent Volume은 남을 수 있다. CRD는 여러 Release가 공유할 수 있어 Chart 삭제 시 자동 제거되지 않는 경우가 일반적이다.

## 자주 발생하는 실패

- `cannot re-use a name`: 같은 Namespace에 Release가 남아 있는지 확인한다.
- `context deadline exceeded`: 단순히 Timeout만 늘리기 전에 Pod Event와 Readiness 실패를 본다.
- Upgrade 뒤 값이 사라짐: `--reuse-values`와 새 Chart Default의 상호작용을 확인한다.
- Rendering은 성공하지만 적용 실패: Cluster API Version과 Admission Webhook 정책을 확인한다.
