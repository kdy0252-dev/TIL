# k6 부하 테스트와 성능 검증

## 1. 개요 (Overview)

k6는 JavaScript로 사용자의 HTTP 흐름과 부하 모델을 정의하고 응답 시간, 실패율, 처리량을 측정하는 부하 테스트 도구입니다. 단순히 많은 요청을 보내는 것이 아니라 다음 질문에 답하는 데 사용합니다.

- 평상시와 Peak Traffic에서 SLO를 만족하는가
- 어느 자원이 먼저 포화되는가
- 부하가 한도를 넘었을 때 점진적으로 느려지는가, 갑자기 붕괴하는가
- 부하가 줄면 자동으로 회복하는가
- 배포 전후 성능이 얼마나 달라졌는가

이 문서의 사례는 차량·배차 업무의 실제 쓰기·조회 흐름과 통계 조회를 k6 Scenario로 재현합니다.

---

## 2. 기능 테스트와 부하 테스트의 차이

기능 테스트는 한 요청의 결과가 맞는지 검증합니다. 부하 테스트는 같은 기능이 동시성과 시간 압력 아래에서도 목표 품질을 유지하는지 검증합니다.

| 관점 | 기능 테스트 | 부하 테스트 |
|---|---|---|
| 주된 질문 | 결과가 올바른가 | 목표 부하에서 지속 가능한가 |
| 동시성 | 낮음 | 실제·Peak·한계 수준 |
| 주요 지표 | Assertion 성공 | P95/P99, 실패율, 처리량, Saturation |
| 실패 원인 | 로직·계약 오류 | Queue, Pool, Lock, CPU, Memory, I/O |
| 실행 시간 | 짧음 | Warm-up과 정상 상태를 포함해 상대적으로 김 |

둘 중 하나로 다른 하나를 대체할 수 없습니다. k6 `check`가 모두 통과해도 P99가 SLO를 넘으면 운영 기준에는 실패입니다.

---

## 3. 사례의 Scenario 구성

사례 구현은 목적별 Profile을 분리합니다.

| Profile | 목적 | 기본 형태 |
|---|---|---|
| Smoke | Script와 인증·Data Setup 확인 | 적은 Iteration |
| Baseline | 평상시 성능 기준선 | 낮은 VU의 지속 부하 |
| Load | 예상 Peak 검증 | 중간 VU의 장시간 유지 |
| Stress | 성능 저하와 포화 지점 탐색 | 단계적으로 VU 증가 |
| Spike | 갑작스러운 Traffic Burst | 빠른 Ramp-up과 짧은 유지 |

Profile을 분리하면 “성공/실패”만 보는 대신 정상 영역, 위험 영역, 붕괴 지점을 구분할 수 있습니다.

---

## 4. 실제 업무 흐름 모델링

단일 `GET /health`를 반복하면 Framework와 Network의 최대 처리량만 측정할 뿐 업무 시스템을 설명하지 못합니다. 사례 Scenario는 다음 흐름을 조합합니다.

```text
회원 생성·조회
  -> 예약 생성·상세·목록
  -> 배차 가능 차량 조회·검증
  -> 배차
  -> 운행 시작·종료
  -> 운행·통계 조회
```

차량, 기사, 차고지, 일정 같은 선행 Resource는 `setup()`에서 만들고 Iteration은 실제 업무 데이터를 생성·조회합니다. 이 방식은 DB Transaction, Lock, Cache, 외부 호출, Projection 조립을 함께 자극합니다.

업무 비율도 중요합니다. 실제 운영이 조회 90%, 쓰기 10%인데 모든 VU가 매 Iteration마다 무거운 쓰기를 수행하면 시스템의 다른 한계를 측정하게 됩니다. 운영 Access Log와 업무량을 근거로 요청 Mix를 정합니다.

---

## 5. Closed Model과 Open Model

VU와 `sleep()`을 사용하는 Scenario는 보통 Closed Model입니다. 응답이 느려지면 VU가 다음 요청을 늦게 보내므로 유입률도 함께 낮아집니다.

```text
Closed Model 처리량 ≈ VU / (응답 시간 + Think Time)
```

실제 외부 요청이 일정한 속도로 도착하는 서비스는 `constant-arrival-rate`나 `ramping-arrival-rate` 같은 Open Model이 더 적합할 수 있습니다. 시스템이 느려져도 도착률을 유지해 Queue 증가와 Backpressure를 관찰할 수 있습니다.

두 모델의 질문은 다릅니다.

- Closed Model: 동시 사용자 N명이 경험하는 성능은 어떤가
- Open Model: 초당 R건이 계속 들어올 때 시스템이 감당하는가

사용자 수를 RPS로 잘못 해석하지 않습니다.

---

## 6. Threshold 설계

Threshold는 테스트 결과를 자동 판정하는 계약입니다.

```javascript
export const options = {
  thresholds: {
    http_req_failed: ['rate<0.01'],
    http_req_duration: ['p(95)<500', 'p(99)<1000'],
  },
};
```

전체 HTTP Metric 하나만 보면 빠른 Health Check가 느린 업무 API를 가릴 수 있습니다. Tag로 Endpoint Group을 나누고 핵심 Journey별 Threshold를 둡니다.

```javascript
'http_req_duration{group:booking-write}': ['p(95)<800'],
'http_req_duration{group:dashboard-read}': ['p(95)<500'],
'http_req_failed{group:critical}': ['rate<0.005'],
```

Threshold 값은 희망 숫자가 아니라 Product SLO, 기존 Baseline, 사용자 허용 시간에서 도출합니다. Stress와 Spike는 일부 실패를 허용하더라도 시스템이 회복해야 한다는 별도 기준이 필요합니다.

---

## 7. Check와 업무 정합성

HTTP 200만 확인하면 오류 Payload나 잘못된 상태 전이를 놓칩니다.

```javascript
check(response, {
  'created': (result) => result.status === 201,
  'has id': (result) => result.json('id') !== undefined,
});
```

쓰기 Scenario는 후속 조회에서 생성 결과를 확인합니다. 다만 모든 응답 Body를 Parsing하면 부하 발생기 CPU가 병목이 될 수 있으므로 핵심 Assertion만 유지합니다.

Check 실패율과 HTTP 실패율은 별도 Metric입니다. HTTP 200인데 업무 Check가 실패한 경우도 Test 실패로 판정해야 합니다.

---

## 8. Test Data 격리

동시 VU가 같은 계정·차량·일정을 수정하면 Application 성능이 아니라 Test Data 충돌을 측정할 수 있습니다. Run ID, VU ID, Iteration ID로 고유 데이터를 만듭니다.

```text
load-<run-id>-<vu-id>-<iteration-id>
```

사례 구현은 부하 테스트 전용 업무 단위와 실행별 계정을 사용하고 파괴적 Teardown을 피합니다. 이 방식은 공유 개발 환경을 보호하지만 반복 실행할수록 데이터가 누적되는 단점이 있습니다.

개선 방법은 다음과 같습니다.

- 부하 테스트 전용 Tenant·Namespace·DB를 사용합니다.
- 생성 데이터에 Run ID와 TTL을 기록합니다.
- 테스트 종료 후 비동기 Cleanup Job을 실행합니다.
- Cleanup은 해당 Run의 데이터만 삭제하고 멱등하게 만듭니다.
- 운영 환경 실행을 Code와 IAM 양쪽에서 차단합니다.

---

## 9. 인증과 Secret

Token과 Test Account Credential을 Script·README 기본값으로 넣지 않습니다. 공유 환경에서는 Secret Manager나 CI Secret을 통해 Runtime Environment Variable로 주입합니다.

부하 테스트 계정은 다음 권한만 가집니다.

- 전용 Tenant의 Test Data 접근
- 필요한 API Operation
- 운영 사용자·설정 접근 금지
- 짧은 만료와 실행 후 회수

Test 결과 JSON과 Error Log에도 Authorization Header나 개인정보가 남을 수 있으므로 Artifact Masking과 Retention을 적용합니다.

---

## 10. Client가 병목인지 확인하기

부하 발생기 CPU, Network, Socket이 포화되면 서버가 아니라 k6의 한계를 측정합니다. 다음을 함께 확인합니다.

- 부하 발생기 CPU와 Memory
- Dropped Iteration
- `vus_max`와 실행 가능한 VU
- Network 대역폭과 DNS
- k6와 대상 사이의 물리적 위치

큰 부하는 여러 Generator로 분산하고 시계를 동기화합니다. Laptop과 Docker Desktop 결과는 개발 Baseline에는 쓸 수 있지만 운영 Capacity의 절대값으로 사용하기 어렵습니다.

---

## 11. 서버 지표와 상관 분석

k6 결과만으로 병목 원인을 확정할 수 없습니다. 같은 시간축에 다음 지표를 수집합니다.

| 계층 | 주요 지표 |
|---|---|
| HTTP | RPS, P95/P99, 4xx·5xx, Active Request |
| JVM | CPU, Heap, GC Pause, Virtual Thread Pinning |
| DB | Pool Active·Pending, Query P95, Lock, Connection |
| Redis | Command Latency, Hit Ratio, Connection, Eviction |
| Kubernetes | CPU Throttling, Memory, Restart, HPA, Pending Pod |
| 비동기 | Queue Length, Oldest Age, Retry, 처리율 |

사례 구현은 Docker Container CPU·Memory를 별도 수집해 k6 HTTP 결과와 Report로 합칩니다. 운영 EKS에서는 Prometheus와 Trace를 같은 Test Run ID·시간 범위로 연결하는 편이 좋습니다.

---

## 12. 병목 판정 예시

### DB Pool 포화

RPS가 더 늘지 않고 Pool Pending과 응답 시간이 함께 증가합니다. Virtual Thread 수나 Pod를 늘리기 전에 Query, Index, Pool과 DB Connection Budget을 봅니다.

### CPU Throttling

CPU 사용률이 Limit 근처이고 Throttling과 P99가 증가합니다. Request/Limit과 실제 Core 필요량을 조정합니다.

### Lock 경합

특정 쓰기 API만 느려지고 DB Lock Wait가 증가합니다. Transaction 범위, 동일 Row 갱신, Index를 확인합니다.

### Downstream 포화

외부 API Timeout, Circuit Breaker Open, Retry가 증가합니다. 무작정 Retry를 늘리지 말고 Bulkhead와 Load Shedding을 검증합니다.

---

## 13. Baseline과 회귀 비교

서로 다른 환경과 데이터 상태의 결과를 단순 비교하면 안 됩니다. 다음 조건을 고정합니다.

- Application Image Digest
- Infrastructure와 Resource 설정
- DB Snapshot과 데이터 규모
- k6 Script Commit과 Profile
- 부하 발생기 사양·위치
- Warm-up, 실행 시간, 시간대

결과에는 P50, P95, P99, Max뿐 아니라 RPS, 실패율, Resource Peak와 Test 조건을 저장합니다. PR마다 큰 Load Test를 실행하기 어렵다면 Smoke를 CI에, Baseline을 Nightly에, Stress·Spike를 Release 전이나 정기 Capacity Test에 배치합니다.

---

## 14. 현재 사례의 강점

- Smoke, Baseline, Load, Stress, Spike Profile이 분리되어 있습니다.
- 실제 연간·일간 업무량을 근거로 기본 VU를 정했습니다.
- 단순 Endpoint가 아니라 회원→예약→배차→운행 업무 흐름을 실행합니다.
- P95, P99, 실패율 Threshold가 있습니다.
- Local과 개발 환경, Gateway와 Application 직접 호출을 구분합니다.
- HTTP 결과와 Container CPU·Memory를 함께 Report할 수 있습니다.

---

## 15. 현재 사례의 부족한 점

### Open Model 부재

대부분 VU 기반 Closed Model이므로 서버가 느려질 때 실제 도착률도 감소합니다. API별 예상 RPS가 확보되면 Arrival-rate Scenario를 추가해야 Queue와 Load Shedding 한계를 더 정확히 볼 수 있습니다.

### 전역 Threshold 중심

전체 `http_req_duration` Threshold는 빠른 API가 느린 핵심 API를 가릴 수 있습니다. 업무 Group과 Endpoint별 Threshold, Check 실패 Threshold가 필요합니다.

### 데이터 누적

파괴적 Teardown을 피하는 대신 반복 실행 데이터가 남습니다. 전용 Tenant와 Run ID Cleanup, DB 크기 변화 통제가 필요합니다.

### Credential 기본값

Test 계정과 Credential이 문서·설정의 기본값으로 존재할 수 있는 구조는 제거해야 합니다. 이미 노출 가능성이 있다면 먼저 회전하고 CI Secret으로 옮깁니다.

### 자동화된 서버 지표 연결

로컬 Docker 통계는 수집하지만 EKS 실행 시 Prometheus Snapshot, Trace, Pod Event, HPA 변화를 동일 Run ID로 자동 보존하는 단계가 더 필요합니다.

### 회복성 판정

Stress·Spike 후 응답 시간, Queue, Circuit Breaker가 기준 시간 안에 정상으로 돌아오는지를 별도 Threshold로 검증해야 합니다.

---

## 16. 개선 로드맵

1. Script와 문서의 Credential 기본값을 제거하고 기존 값을 회전합니다.
2. 핵심 Journey와 API Group별 Custom Metric·Threshold를 추가합니다.
3. 예상 RPS 기반 `ramping-arrival-rate` Scenario를 추가합니다.
4. 전용 Tenant와 Run ID Cleanup을 자동화합니다.
5. k6 실행에 Test Run ID를 부여하고 Prometheus·Trace·Log를 연결합니다.
6. CI Smoke, Nightly Baseline, Release Stress/Spike로 실행 계층을 나눕니다.
7. 결과를 이전 Baseline과 자동 비교해 허용 회귀율을 넘으면 실패시킵니다.
8. 과부하 뒤 회복 시간과 데이터 정합성을 검증합니다.

---

## 17. 완료 기준

- [ ] 운영 환경을 Code와 권한으로 실행 금지합니다.
- [ ] Script·문서·Artifact에 Credential과 개인정보가 없습니다.
- [ ] 실제 Traffic 비율과 RPS 근거가 Scenario에 반영됩니다.
- [ ] 핵심 Journey별 P95/P99·실패율 Threshold가 있습니다.
- [ ] k6 Client가 병목이 아님을 Generator Metric으로 확인합니다.
- [ ] HTTP 결과와 JVM·DB·Redis·Kubernetes 지표가 같은 Run으로 연결됩니다.
- [ ] 반복 실행 데이터가 격리·정리되어 Baseline을 오염시키지 않습니다.
- [ ] Stress와 Spike 후 정해진 시간 안에 정상 상태로 회복합니다.
- [ ] Image 또는 Resource 변경 전후 성능 회귀를 자동 판정합니다.
