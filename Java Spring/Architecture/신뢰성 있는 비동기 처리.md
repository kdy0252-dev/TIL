# 신뢰성 있는 비동기 처리: Outbox, Claim, Retry, Retention

## 1. 개요 (Overview)
외부 API, Push, SMS, 통계 반영을 비즈니스 Transaction 안에서 직접 실행하면 외부 장애가 핵심 요청을 실패시키고, DB Commit과 외부 호출 사이에 Dual Write 문제가 생깁니다. 신뢰성 있는 비동기 처리는 **Outbox에 의도를 저장하고 별도 Worker가 전달**하는 구조로 해결합니다.

---

## 2. 상태 흐름

```text
PENDING
  -> PROCESSING (Claim)
      -> SUCCEEDED
      -> RETRYABLE_FAILED -> PENDING
      -> FAILED
```

`PROCESSING` 상태에는 Claim Owner와 Claim 시각을 기록합니다. Worker가 중간에 죽으면 Claim Timeout이 지난 항목을 다시 가져올 수 있습니다.

---

## 3. 원자적 Claim
여러 Instance가 같은 Outbox를 처리할 때 조회와 상태 변경을 분리하면 중복 처리가 발생합니다.

```sql
select id
from notification_outbox
where status = 'PENDING'
order by created_at
for update skip locked
limit :batch_size;
```

Transaction 안에서 선택한 Row를 `PROCESSING`으로 변경합니다. PostgreSQL의 `SKIP LOCKED`는 다른 Worker가 잡은 Row를 기다리지 않고 다음 Row를 처리하게 합니다.

---

## 4. Retry 정책
- 재시도 가능한 오류와 영구 오류를 분류합니다.
- Exponential Backoff와 Jitter를 사용합니다.
- 최대 횟수뿐 아니라 최대 경과 시간도 제한합니다.
- 다음 시도 시각을 저장해 매 Polling마다 실패 Row를 읽지 않습니다.
- 동일 외부 요청이 중복되어도 안전하도록 Idempotency Key를 전달합니다.

---

## 5. Polling과 Scheduling
Polling 주기가 짧으면 지연은 줄지만 DB 부하가 늘어납니다. 처리할 Row가 없을 때의 Query 비용, Batch Size, Worker 수와 외부 API Rate Limit을 함께 조정합니다.

Quartz Cluster나 여러 Pod에서 Scheduler가 실행되더라도 실제 안전성은 Scheduler 단일 실행 가정보다 DB의 원자적 Claim에 두는 것이 좋습니다.

---

## 6. Retention과 관측성
완료 Row를 즉시 삭제하면 감사와 장애 분석이 어렵고, 영구 보관하면 Table과 Index가 비대해집니다. 성공과 실패의 Retention 기간을 분리하고 Batch로 정리합니다.

관측해야 할 지표는 대기 수, 가장 오래된 대기 시간, 처리율, 재시도율, 영구 실패 수, Claim Timeout 복구 수와 Scheduler 마지막 성공 시각입니다.

---

## 7. 실무 사례 적용 관점
이 사례는 SMS, FCM, Cognito Sync, Virtual Number, Metrics Worklog Event에 Outbox·Polling·Claim·Retry·Retention 구조를 사용합니다. 각 통합의 외부 오류 특성과 처리량이 다르므로 Batch Size와 Retry 정책을 독립적으로 둡니다.

Metrics는 EU Outbox를 읽고 Redis Realtime Metric과 일별 집계로 반영합니다. Source Outbox와 Target 처리 상태를 함께 관리해 재처리와 Reconciliation이 가능하도록 합니다.

---

## 8. Transaction 경계
업무 데이터와 Outbox Insert는 같은 Local Transaction에 포함합니다.

```text
BEGIN
  -> Booking 상태 변경
  -> Notification Outbox Insert
COMMIT
```

외부 API 호출은 이 Transaction 밖에서 수행합니다. Transaction 안에서 Network를 기다리면 DB Connection과 Lock을 오래 점유하고, 외부 성공 뒤 DB Rollback 같은 불일치가 생깁니다.

## 9. Claim 구현 방식

### Pessimistic Lock + SKIP LOCKED
여러 Worker가 다른 Row를 병렬 처리하기 좋습니다. Claim Transaction은 짧게 끝내고 외부 호출 중 DB Lock을 유지하지 않습니다.

### Conditional Update

```sql
update outbox
set status = 'PROCESSING',
    claimed_by = :worker,
    claimed_at = now()
where id = :id
  and status = 'PENDING';
```

영향받은 Row가 1개일 때만 Claim 성공입니다. 조회 결과를 믿지 않고 Update 결과로 소유권을 결정합니다.

## 10. Lease와 Fencing
Claim Timeout은 Worker 장애 복구를 가능하게 하지만, 느린 이전 Worker가 Timeout 뒤 늦게 완료할 수 있습니다.

```text
Worker A Claim(version=1)
  -> 오래 멈춤
Lease 만료
Worker B Claim(version=2) -> 성공
Worker A 재개 -> 늦은 결과 기록 시도
```

상태 Update에 Claim Version·Owner 조건을 포함하여 이전 Worker의 결과를 거부합니다. 외부 Side Effect는 Provider Idempotency Key로 중복을 줄입니다.

## 11. 오류 분류

| 오류 | 예 | 정책 |
|---|---|---|
| 일시적 | Timeout, 429, 503 | Backoff 후 재시도 |
| 영구적 입력 | 잘못된 전화번호, 만료 Token | 즉시 FAILED |
| 인증·설정 | Credential 오류 | 짧은 재시도 후 Alert·Circuit Open |
| 내부 데이터 | 참조 누락, Serialization 오류 | 격리 후 수동 분석 |

예외 Class만으로 분류하지 않고 외부 API Status와 Error Code를 안정적인 내부 오류 유형으로 변환합니다.

## 12. 순서 보장
같은 Aggregate의 Event 순서가 중요하면 `aggregate_id`와 Sequence를 저장합니다. 여러 Worker가 같은 Aggregate의 Event를 동시에 처리하지 않게 Partition 또는 Key별 직렬화를 사용합니다.

전체 Global 순서를 보장하려 하면 처리량과 가용성이 크게 떨어집니다. 업무에 필요한 범위의 순서만 보장합니다.

## 13. Poison Message
항상 실패하는 Event가 Queue 앞을 막지 않도록 최대 시도 후 FAILED·Dead Letter 상태로 격리합니다. 원본 Payload, 오류 분류, 마지막 Stack Trace, 시도 횟수와 상관관계 ID를 보관합니다.

수정 후 재처리할 때 원본을 덮어쓰지 않고 재처리 이력과 Actor를 남깁니다.

## 14. Polling 용량 계산

```text
최대 처리량 ≈ Worker 수 × Batch Size / 평균 처리 시간
```

외부 API Rate Limit, DB Pool과 Thread 수가 실제 상한입니다. Batch를 크게 하면 처리량은 늘 수 있지만 한 Transaction, Memory와 실패 복구 단위가 커집니다.

## 15. 테스트 전략
- 같은 Row를 여러 Worker가 Claim해 한 명만 성공하는지 확인합니다.
- Worker가 Claim 후 죽었을 때 Timeout 뒤 복구되는지 확인합니다.
- 느린 이전 Worker의 결과가 Fencing되는지 확인합니다.
- Retryable·Permanent 오류 분류를 Contract Test로 검증합니다.
- 동일 Event를 두 번 처리해 Side Effect가 중복되지 않는지 확인합니다.
- Retention Job이 진행 중 Row를 삭제하지 않는지 확인합니다.

---

## 16. 실무 사례 적용 진단과 개선 과제

이 사례에는 Cognito Sync, Virtual Number, Driving Route Address, Worklog Metric 등 여러 Outbox·Claim·Retry 구현이 존재합니다. 개별 흐름에는 Processing Timeout과 재시도가 있지만 상태 이름, Backoff, 최대 시도, Poison Message 처리와 운영 Metric이 흐름마다 다릅니다.

가장 큰 공백은 공통 운영 계약입니다. 모든 비동기 흐름에 `PENDING/PROCESSING/RETRYABLE_FAILED/FAILED/COMPLETED`, `claimed_until`, `retry_count`, `next_retry_at`, 실패 분류와 Correlation ID를 공통 기준으로 적용하고, 강제 재처리 Tool과 Runbook을 제공합니다.

완료 기준은 Worker 강제 종료·외부 5xx·DB 저장 실패 Test에서 유실 없이 회복하고, Oldest Event Age·Retry·Terminal Failure가 Dashboard/Alert에 보이며, 운영자가 ID 하나로 원 요청부터 외부 결과까지 추적할 수 있는 상태입니다.
