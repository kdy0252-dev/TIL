# 차량·배차 플랫폼 소프트웨어 아키텍처 사례

## 1. 시스템 개요 (System Context)
이 문서의 사례는 차량·회원·예약·배차·운행·결제·통계를 다루는 Multi-tenant Backend입니다. 특정 사내 프로젝트를 알아야 이해할 수 있는 설명이 아니라, 비슷한 업무 시스템에 재사용할 수 있는 아키텍처 판단을 정리합니다. 애플리케이션은 기능과 배포 책임에 따라 다음 실행 단위로 나뉩니다.

```text
Client
  -> Gateway
      ├─ Core Domain Application
      ├─ BFF
      └─ Metrics

Supporting Runtime
  ├─ PostgreSQL
  ├─ Redis
  ├─ Cognito / FCM / Map / SMS
  └─ Observability

One-off Runtime
  └─ Legacy Migration
```

---

## 2. 핵심 아키텍처 스타일

| 스타일 | 적용 목적 |
|---|---|
| Modular Monolith | 하나의 배포 단위 안에서 업무 모듈 경계 유지 |
| Vertical Slice | 회원·예약·운행 등 기능 단위로 코드 응집 |
| Hexagonal Architecture | Domain과 Web·DB·외부 API 기술 분리 |
| Domain-Driven Design | Aggregate와 Value Object에 업무 규칙 집중 |
| CQRS | 상태 변경 모델과 복잡한 조회·통계 모델 분리 |
| Event·Outbox | 외부 알림과 통계를 신뢰성 있게 비동기 처리 |
| Schema Multi-tenancy | Tenant별 데이터 격리와 공용 데이터 분리 |

한 가지 아키텍처를 전체에 획일적으로 적용하지 않고, 각 문제에 맞는 스타일을 조합합니다.

---

## 3. Slice 내부 구조

```text
{slice}
  ├─ adapter
  │   ├─ in/web
  │   └─ out/persistence | external
  └─ application
      ├─ port
      │   ├─ in
      │   └─ out
      └─ domain
          ├─ model
          ├─ model/vo
          └─ service
```

의존성은 Adapter에서 Application Core 방향으로 향합니다. Domain은 Spring MVC, JPA Entity, Firebase SDK 같은 외부 기술을 알지 않습니다.

---

## 4. 데이터 흐름

### Command
```text
Request -> Command -> Use Case -> Domain Model -> Out Port -> Persistence Adapter
```

### Query
```text
Request -> Query -> Query Out Port -> Projection / jOOQ / QueryDSL -> Resource
```

조회는 모든 경우에 Aggregate를 Hydration하지 않고 목적에 따라 Projection을 사용합니다. 상태 변경은 Aggregate를 Reconstitution한 뒤 비즈니스 메서드를 호출합니다.

---

## 5. 모듈 간 통신
- 같은 Slice 내부 구현은 Package-private과 명확한 Package 구조로 감춥니다.
- 다른 Slice는 `application.port.in.method`와 `@NamedInterface`로 공개된 계약만 사용합니다.
- 상세 Service나 Persistence Adapter를 다른 Slice에서 직접 호출하지 않습니다.
- 비동기 결과가 적합한 경우 Event·Outbox로 시간적 결합을 낮춥니다.

---

## 6. 구조 검증
Spring Modulith, jMolecules, ArchUnit과 프로젝트 Architecture Test가 의존성 방향을 검증합니다.

```text
Code Change
  -> Compiler / Static Analysis
  -> Unit / Integration Test
  -> Architecture Test
  -> API E2E
```

문서에 적힌 규칙을 CI에서 실행 가능한 제약으로 바꾸어 구조적 회귀를 막습니다.

---

## 7. 주요 품질 속성
- **변경 용이성**: Slice와 Port로 변경 범위를 제한합니다.
- **신뢰성**: Outbox, Retry, Idempotency, Reconciliation을 사용합니다.
- **확장성**: Virtual Thread, Redis, Batch, Query 전용 기술을 조합합니다.
- **격리**: Tenant Schema, Bulkhead, Readiness와 Module Boundary를 사용합니다.
- **관측성**: Metric, Trace, Structured Log와 Correlation ID를 연결합니다.
- **배포 안전성**: GitOps와 Argo Rollouts로 선언적·점진적 배포를 수행합니다.

---

## 8. Domain 모델링
Aggregate는 Transaction으로 함께 일관성을 지켜야 하는 최소 경계입니다. 모든 연관 Entity를 하나의 큰 Aggregate로 만들지 않습니다.

- `create()`: 신규 Identity와 초기 상태
- `load()`·`reconstitute()`: 저장 상태 복원
- 의미 있는 Business Method: 상태 변경
- Value Object: 단위·검증·불변성

Setter와 Persistence Entity 노출을 피하고 Domain이 업무 상태 전이를 소유합니다.

## 9. 오류 처리
Port 경계의 실패는 `Either` 등 명시적인 오류 값으로 표현하고, Domain Service 내부의 정상 업무 흐름은 기술 오류 타입에 결합하지 않습니다.

```text
Persistence Exception
  -> Adapter Error 변환
  -> Out Port Either
  -> Application 업무 분기
  -> Web Error 계약
```

## 10. 일관성 경계
- Aggregate 내부: 강한 일관성
- 같은 DB의 여러 Aggregate: Application Transaction을 신중히 사용
- 외부 시스템·후속 처리: Outbox와 최종 일관성
- 통계 Read Model: Event·Batch로 지연 허용

업무 요구가 허용하는 가장 작은 강한 일관성 범위를 선택합니다.

## 11. 조회 아키텍처
간단한 Aggregate 조회는 JPA, 동적 검색은 RSQL·QueryDSL, 복잡한 통계는 jOOQ를 사용합니다. 기술 선택은 Module별 유행이 아니라 Query 특성에 따릅니다.

Projection과 Resource는 Read 요구에 맞추고 Domain Model을 화면 DTO로 사용하지 않습니다.

## 12. 비동기 경계
SMS, FCM, Cognito, Metrics처럼 외부 실패와 재처리가 필요한 작업은 Outbox로 분리합니다. Scheduler 단일 실행보다 원자적 Claim과 멱등성을 신뢰성의 기반으로 둡니다.

## 13. Legacy Migration
Migration Runner는 Online Application과 다른 수명 주기·권한·실패 정책을 가집니다. Source→Target ID Mapping, Upsert, Checkpoint와 Smoke Check로 재실행 가능하게 만듭니다.

## 14. Gateway와 BFF
Gateway는 인증·라우팅·Cross-cutting 경계이며 업무 세부 권한을 소유하지 않습니다. BFF는 Client별 조합과 표현을 담당하고 Core Domain 규칙을 복제하지 않습니다.

## 15. Metrics 분리
Metrics는 Online Transaction의 복잡한 통계 부하를 분리합니다. Source Outbox와 Tenant별 Query를 사용해 Realtime Redis와 일별 집계를 구성합니다.

이는 물리적으로 분리된 CQRS Read Side에 가까우며 Eventual Consistency와 재집계 정책이 필요합니다.

## 16. 아키텍처 Decision 기준
새 기술·Module을 추가할 때 다음을 기록합니다.

- 해결하려는 품질 속성
- 기존 방식의 한계
- 선택한 대안과 거절한 대안
- 데이터·Transaction·운영 경계
- 실패와 Rollback
- 검증 방법

## 17. 주요 위험
- Named Interface가 늘어 Slice가 서로 강결합
- 공용 `global` Package 비대화
- Projection이 쓰기 경계를 우회
- Scheduler가 중복 실행되지 않을 것이라는 가정
- Shared EKS·DB의 Noisy Neighbor
- Configuration과 Secret의 저장소 노출

## 18. 진화 방향
모듈 경계와 데이터 소유권을 먼저 강화하고, 독립 배포의 실익이 확인된 Slice만 Service로 분리합니다. 분산 시스템 비용을 미리 지불하지 않으면서 분리 가능한 구조를 유지하는 것이 목표입니다.

---

## 19. 사례 구현 진단과 개선 로드맵

현재 핵심 업무 애플리케이션은 Vertical Slice, Hexagonal Port, ArchUnit·Spring Modulith 검사를 갖추고 있어 구조적 기반은 좋습니다. 다만 공개 Port를 Orchestrator Package로 정돈하려는 Architecture Test TODO가 남아 있고, 일부 최상위 Service에는 조회 조립과 세부 규칙이 함께 존재합니다.

부족한 점은 **경계 규칙의 목표 상태가 코드와 문서 사이에서 아직 완전히 일치하지 않는 것**입니다. 외부 Slice가 어떤 Inbound Contract만 사용할 수 있는지, Transaction과 Event 경계가 어디인지 일관된 판정 기준이 더 필요합니다.

해결 순서는 ① Slice별 공개 Contract Inventory 작성 ② 직접 구현 참조를 ArchUnit으로 차단 ③ Controller-facing Service를 업무 흐름만 남기도록 분리 ④ Cross-slice 변경을 동기 호출·Event 중 하나로 명시하는 것입니다. `NamedInterfaceArchTest`의 TODO가 제거되고 새 위반이 CI에서 차단되며 핵심 Slice의 의존 Graph에 Cycle이 없을 때 완료로 봅니다.
