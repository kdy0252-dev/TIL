# Spring Batch Flow Step

Spring Batch Job은 하나 이상의 Step을 순서대로 실행한다. 모든 Job이 `step1 -> step2 -> step3`처럼 직선으로 끝나지는 않는다. 검증 결과에 따라 다른 Step으로 이동하거나, 운영자 확인을 위해 멈췄다가 재시작해야 할 수 있다. **Flow**는 이 실행 경로와 전이 조건을 표현한다.

## BatchStatus와 ExitStatus

두 상태를 구분해야 Flow 조건을 이해할 수 있다.

| 상태 | 의미 |
|---|---|
| BatchStatus | Framework가 관리하는 실행 상태: STARTED, COMPLETED, FAILED 등 |
| ExitStatus | Step이 다음 전이를 위해 반환하는 논리적 결과 Code |

`on()`은 `BatchStatus`가 아니라 `ExitStatus.exitCode`와 Pattern Matching한다. Step은 실행 자체는 성공했지만 업무 결과에 따라 사용자 정의 ExitStatus를 반환할 수 있다.

```java
contribution.setExitStatus(new ExitStatus("NO_DATA"));
```

## FlowStep
### Transition - on() / to() / stop() ,fail(), end(), stopAndRestart()
#### Transition
- Flow 내 Step의 조건부 전환(전이) 를 정의함
- Job의 API설정에서 on(String pattern) 메소드를 호출하면 TransitionBuilder가 반환되어  
    Transition Flow를 구성할 수 있음
- Step의 종료상태(ExitStatus) 가 어떤 pattern 과도 매칭이되지않으면 스프링 배치에서 예외를 발생하고 Job은 실패
- transition 은 구체적인 것 부터 그렇지 않은 순서로 적용된다

```text
NO_DATA -> notifyStep
FAILED  -> recoveryStep
*       -> nextStep
```

마지막 `*`는 예상하지 못한 정상 결과를 처리하는 안전망이 될 수 있지만, 오류까지 무조건 성공 경로로 보내지 않게 구체적 Pattern을 먼저 정의한다.
### API
on
- Step의 실행결과로 돌려받는 종료 상태(ExitStatus) 와 매칭하는 패턴 스키마 , BatchStatus 와 매칭하는것이 아님
- pattern 과 ExitStatus가 매칭이되면 다음으로 실행할 Step을 지정할 수 있다.
- 특수문자는 2가지만 허용
- * = 0 개이상의 모든 문자
- ? = 1개의 문자
to  = 다음으로 실행할 단계를 지정
from = 이전 단계에서 정의한 Transition 을 새롭게 추가 정의함
#### Job을 중단하거나 종료하는 Transition API
- Flow가 실행되면 FlowExecutionStatus 에 상태값이 저장되고 최종적으로 Job의 BatchStatus 와 ExitStatus 에 반영된다.
- Step의 BatchStatus 및 ExitStatus 에는 아무런 영향을 주지 않고 Job의 상태만을 번경한다.

stop
- FlowExecutionStatus 가 stopped 상태로 종료되는 transition
- Job의 BatchStatus 와 ExitStatus 가 stopped 으로 종료됨
fail
- FlowExecutionStatus가 failed 상태로 종료되는 transition
- Job의 BatchStatus 와 ExitStatus 가 failed 으로 종료됨
end
- FlowExecutionStatus가 completed 상태로 종료되는 transition
- Job의 BatchStatus 와 ExitStatus 가 completed 으로 종료됨
- Step의 ExitStatus가 failed 이더라도 job의 batchstuats는 completed 로 종료하도록 가능하며  
    이때 Job의 재시작은 불가능함
stopAndRestart
- stop과 기본흐름은 동일
- 특정 step 에서 작업을 중단하도록 설정하면 중단 이전의 Step만 completed 로 저장되고 이후의 step은 실행되지 않고 stopped 상태로 job 종료
- job이 다시 실행됐을 때 실행해야할 Step을 restart 인자로 넘기면 이전에 completed 로 저장된 step 은 건너뛰고 중단이후 step 부터 시작한다.

## stop, fail, end의 의미 차이

- `stop()`은 운영 개입 후 같은 JobInstance를 재시작할 수 있는 중단이다.
- `fail()`은 Job을 실패로 끝내며 원인을 해결한 뒤 재시작할 수 있다.
- `end()`는 현재 Flow를 성공 완료로 바꾼다. 앞 Step이 실패했더라도 JobInstance가 COMPLETED가 되면 같은 Parameter로 재시작할 수 없으므로 신중해야 한다.

실패를 감추기 위해 `end()`를 사용하면 Monitoring과 재처리가 어려워진다. “업무적으로 더 처리할 Data가 없음”처럼 성공으로 해석 가능한 상태에만 사용한다.

예시 코드
```Java title="Flow Step 예시 코드"
@Bean
public Job batchJob() {
    return new JobBuilder("batchJob", jobRepository)
        .start(step1())
        .on("COMPLETED").to(step2())
        .from(step1())
        .on("FAILED").to(step3())
        .end()
        .build();
}
```

Spring Batch 5에서는 `JobBuilderFactory` 대신 `JobRepository`를 받는 Builder를 직접 사용하는 구성이 일반적이다.

## Decider가 필요한 경우

Step의 ExitStatus만으로 분기하기 어렵고 Job Parameter, 시간 또는 여러 실행 결과를 함께 봐야 한다면 `JobExecutionDecider`를 사용한다.

```java
public FlowExecutionStatus decide(
    JobExecution jobExecution,
    StepExecution stepExecution
) {
    boolean dryRun = jobExecution.getJobParameters()
        .getString("mode", "RUN")
        .equals("DRY_RUN");
    return new FlowExecutionStatus(dryRun ? "SKIP_WRITE" : "WRITE");
}
```

Decider에 업무 처리 전체를 넣지 않고 경로 선택에 필요한 판단만 둔다.

## Restart 가능성을 설계한다

Flow가 재시작될 때 COMPLETED Step은 기본적으로 다시 실행되지 않는다. Step이 외부 File, Message나 API에 Side Effect를 만들었다면 재실행 여부와 멱등성을 명확히 해야 한다.

- Job Parameter가 같은 실행은 같은 JobInstance로 식별된다.
- 실패 Step의 ExecutionContext가 재시작 위치를 저장할 수 있다.
- `allowStartIfComplete`와 `startLimit`은 의도 없이 켜지 않는다.
- Flow 변경 후 과거 JobInstance를 재시작할 때 경로 호환성을 확인한다.

## Test

Happy Path만 Test하면 Transition Pattern 오류를 놓친다. 각 ExitStatus에 대해 다음 Step, 최종 BatchStatus와 재시작 가능 여부를 검증한다. Job Repository를 사용하는 통합 Test로 실제 Metadata 저장까지 확인한다.

## 기억할 점

Flow는 Step 목록을 보기 좋게 연결하는 문법이 아니다. 실패, 중단과 재시작을 포함한 Batch State Machine이다. ExitStatus의 업무 의미와 최종 Job 상태를 먼저 정의한 뒤 Transition API를 선택해야 한다.
