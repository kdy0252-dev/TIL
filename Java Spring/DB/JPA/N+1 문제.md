# JPA N+1 문제

N+1은 부모 N개를 한 Query로 조회한 뒤 각 부모의 연관 관계를 읽을 때 추가 Query가 N번 발생하는 문제다. `EAGER`만의 문제가 아니라 Transaction 안에서 Lazy Collection을 순회해도 발생한다.

```text
SELECT booking ...        -- 1회
SELECT passenger ...      -- booking마다 N회
```

## Entity Mapping

연관 관계는 기본적으로 Lazy로 두고 Use Case별 조회 전략을 Query에서 선택한다.

```java
@Entity
@Table(name = "booking")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class BookingJpaEntity {

    @Id
    private Long id;

    @OneToMany(
        mappedBy = "booking",
        fetch = FetchType.LAZY,
        cascade = CascadeType.ALL,
        orphanRemoval = true
    )
    private final List<PassengerBookingJpaEntity> passengers = new ArrayList<>();
}
```

`EAGER`로 바꾸면 모든 Use Case가 Collection을 항상 필요로 한다고 가정하게 되고, 다른 연관 관계와 Cartesian Product를 만들 수 있다.

## Fetch Join

부모 수가 제한되고 Collection 하나가 필요한 상세 조회에 적합하다.

```java
public interface BookingJpaRepository extends JpaRepository<BookingJpaEntity, Long> {

    @Query("""
        select distinct booking
        from BookingJpaEntity booking
        left join fetch booking.passengers
        where booking.id = :bookingId
        """)
    Optional<BookingJpaEntity> findByIdWithPassengers(@Param("bookingId") long bookingId);
}
```

Collection Fetch Join과 Pagination을 함께 쓰면 Row 기준 Limit 때문에 Memory Pagination이나 잘못된 결과가 생길 수 있다. 여러 To-many를 동시에 Fetch Join하면 Row 수가 곱으로 증가한다.

## Two-step Pagination

먼저 부모 ID Page를 조회하고, 두 번째 Query로 필요한 관계를 Batch 조회한다.

```java
@Query("""
    select booking.id
    from BookingJpaEntity booking
    where booking.centerId = :centerId
    order by booking.serviceDate desc, booking.id desc
    """)
Page<Long> findPageIds(@Param("centerId") long centerId, Pageable pageable);

@Query("""
    select distinct booking
    from BookingJpaEntity booking
    left join fetch booking.passengers
    where booking.id in :bookingIds
    """)
List<BookingJpaEntity> findAllWithPassengersByIdIn(@Param("bookingIds") List<Long> bookingIds);
```

Adapter는 Page ID 순서를 복원한다.

```java
@Override
@Transactional(readOnly = true)
public Page<BookingListResource> findPage(long centerId, Pageable pageable) {
    Page<Long> idPage = repository.findPageIds(centerId, pageable);
    Map<Long, BookingJpaEntity> bookingById = repository.findAllWithPassengersByIdIn(idPage.getContent())
                                                           .stream()
                                                           .collect(Collectors.toMap(
                                                               BookingJpaEntity::getId,
                                                               Function.identity()
                                                           ));

    List<BookingListResource> content = idPage.getContent()
                                                  .stream()
                                                  .map(bookingById::get)
                                                  .filter(Objects::nonNull)
                                                  .map(mapper::toListResource)
                                                  .toList();
    return new PageImpl<>(content, pageable, idPage.getTotalElements());
}
```

## DTO Projection

화면이 Aggregate 전체가 아니라 일부 Column만 필요하면 Projection이 더 직접적이다.

```java
public interface BookingPassengerRow {
    Long getBookingId();
    LocalDate getServiceDate();
    Long getPassengerId();
    String getPassengerName();
}

@Query("""
    select booking.id as bookingId,
           booking.serviceDate as serviceDate,
           passenger.id as passengerId,
           passenger.name as passengerName
    from BookingJpaEntity booking
    left join booking.passengers passenger
    where booking.id in :bookingIds
    order by booking.serviceDate desc, booking.id desc, passenger.id asc
    """)
List<BookingPassengerRow> findPassengerRows(@Param("bookingIds") List<Long> bookingIds);
```

Projection Row는 Adapter에서 `groupingBy`로 Resource 구조로 조립한다. Query Model을 Domain Aggregate 복원에 재사용하지 않는다.

## Batch Fetch

`@BatchSize` 또는 `hibernate.default_batch_fetch_size`는 여러 Lazy Proxy/Collection을 `IN` Query로 묶어 N번을 줄인다. Query 수를 완전히 1개로 만드는 기능은 아니며 Batch 크기, Parameter Limit과 Persistence Context 크기를 측정해야 한다.

```yaml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

## 검증

SQL Log를 눈으로 보는 Test보다 Query 수를 Assert한다.

```java
@Test
void loadsBookingPageWithoutNPlusOneQueries() {
    queryCount.clear();

    Page<BookingListResource> page = adapter.findPage(centerId, PageRequest.of(0, 20));

    assertThat(page.getContent()).hasSize(20);
    assertThat(queryCount.select()).isLessThanOrEqualTo(3);
}
```

Test Data가 부모 하나뿐이면 N+1을 발견하지 못한다. 관계가 있는 부모 여러 개와 빈 Collection을 함께 넣는다.

## 기억할 점

N+1 해결은 모든 관계를 Fetch Join하는 것이 아니다. Use Case별로 상세 조회는 Fetch Join, 목록은 ID Pagination과 Batch, 읽기 전용 화면은 Projection을 선택하고 실제 Query 수와 Row 수를 검증해야 한다.
