# 실무 프로젝트 기술 스택 갭 분석

## 1. 이 문서의 목적

이 문서는 차량·배차 도메인의 Spring 기반 Backend와 EKS 기반 배포 플랫폼을 사례로 삼아 다음 질문에 답합니다.

- 실제로 어떤 기술과 아키텍처 패턴이 사용되는가
- 현재 구현 수준에서 무엇이 잘 갖춰져 있는가
- 어떤 부분이 운영·보안·확장성 측면에서 부족한가
- 부족한 부분을 어떤 순서로 해결해야 하는가
- 무엇을 확인해야 개선이 끝났다고 판단할 수 있는가

특정 저장소의 내부 이름과 자격 증명, 사내 URL은 기록하지 않습니다. 구현 클래스명도 기술 판단의 근거가 되는 경우에만 일반화해 설명합니다.

- Private EKS와 환경별 Namespace

## 4. 잘 갖춰진 부분

### 구조적 경계

- 업무 Slice 내부를 Adapter, Port, Domain으로 분리합니다.
- ArchUnit, jMolecules, Spring Modulith로 의존 방향을 검사합니다.
- 복잡한 조회에는 Projection, QueryDSL, jOOQ를 목적에 따라 선택합니다.
- OSIV를 끄고 조회 조립과 Domain Hydration을 명시적으로 수행합니다.

### 실패 대응 기반

- 외부 Side Effect에 Outbox, Claim, Retry, Retention을 사용합니다.
- DB 과부하를 Readiness와 Load Shedding에 연결합니다.
- 외부 지도 API에는 Timeout, Circuit Breaker, Rate Limit이 있습니다.
- Liveness, Readiness, Startup Probe와 Graceful Shutdown을 적용합니다.

### 배포와 관측성

- 검증한 Image를 ECR에 올리고 GitOps로 배포합니다.
- Blue-Green의 Active·Preview Service를 분리합니다.
- Metric, Log, Trace와 Synthetic Check를 함께 사용합니다.
- Infrastructure와 Application Desired State의 소유권을 분리합니다.

## 5. 최우선 개선 과제

### P0. 비밀 정보의 저장소 노출 제거

환경별 Application 설정과 Infrastructure 입력에 장기 Credential이나 Password가 직접 들어갈 수 있는 구조가 확인됩니다. Private Repository도 Secret 저장소가 아니며 Git History와 Terraform State에 값이 남을 수 있습니다.

해결 방법:

1. 노출 가능성이 있는 Credential을 먼저 회전합니다.
2. Cloud Secret Manager를 Source of Truth로 정합니다.
3. Workload Identity(IRSA)로 대체 가능한 Access Key를 제거합니다.
4. Git에는 Secret 이름과 참조만 저장합니다.
5. Secret Scanner를 Pre-commit과 CI에 추가합니다.
6. Terraform Plan, Log, Rendered Manifest의 Masking을 검증합니다.

완료 기준은 Git 전체 History와 CI Artifact에 재사용 가능한 Secret이 없고, Rotation 후 서비스가 무중단으로 동작하는 상태입니다.

### P0. 운영 데이터 계층의 단일 AZ 위험

삭제 방지와 Storage 암호화는 적용됐지만 운영 RDS가 Multi-AZ가 아닌 구성이 확인됩니다. Instance 또는 AZ 장애 시 자동 Failover가 보장되지 않습니다.

해결 방법:

1. 실제 RTO·RPO를 정의합니다.
2. 비운영 환경에서 Multi-AZ 전환과 강제 Failover를 시험합니다.
3. Connection Pool의 DNS 갱신과 재연결을 검증합니다.
4. Backup에서 새 Instance를 복구하고 시간을 측정합니다.
5. Redis Primary 교체와 Cache 전체 유실도 함께 시험합니다.

완료 기준은 RDS·Redis 장애 주입 후 목표 시간 안에 복구하고 데이터 정합성을 확인한 상태입니다.

### P0. 운영 Endpoint 접근 통제

Gateway의 Actuator Role 기반 접근 제어가 TODO로 남아 있습니다. 관리 Endpoint는 Application 정보와 Metric을 노출할 수 있으므로 Network 위치만 신뢰해서는 안 됩니다.

해결 방법은 Load Balancer에 필요한 Liveness만 최소 공개하고, Prometheus Endpoint는 Cluster 내부 경로로 제한하며, 나머지는 운영 Role과 감사 Log를 요구하는 것입니다.

완료 기준은 외부·일반 사용자 접근 Test가 401/403으로 실패하고 Monitoring 수집만 정상인 상태입니다.

## 6. 높은 우선순위 개선 과제

| 영역 | 현재 부족한 점 | 해결 방향 | 완료 판정 |
|---|---|---|---|
| Workload 가용성 | Application PDB가 확인되지 않음 | SLO 기반 PDB와 Topology Spread | Node Drain 중 최소 Replica 유지 |
| 환경 격리 | NetworkPolicy·Quota·LimitRange가 확인되지 않음 | Default-deny와 Namespace Resource 경계 | 환경 간 임의 통신·자원 침범 차단 |
| NAT 가용성 | 단일 NAT Gateway | SLO에 따라 AZ별 NAT, VPC Endpoint | NAT/AZ 장애 Game Day 통과 |
| 비동기 처리 | Outbox별 상태·Backoff·Metric 불일치 | 공통 상태 머신과 운영 계약 | 유실 없는 재시도와 Oldest Age Alert |
| 멱등성 | 외부 성공 후 내부 저장 실패의 중복 가능성 | Idempotency Key, Unique Constraint, Reconciliation | 동일 요청 N회 결과 1개 |
| Architecture | 공개 Port 전환 TODO와 예외 | Named Interface 계약과 예외 만료 | 신규 경계 위반 CI 차단 |
| Multi-tenancy | Async·Connection Pool의 Tenant Context 누수 위험 | Context 명시화와 교차 Tenant Test | 모든 누수 시나리오 차단 |
| Progressive Delivery | Health 중심 Promotion | SLO·Synthetic·DB 호환 Analysis | 회귀 시 전환 차단·자동 Abort |

## 7. 중간 우선순위 개선 과제

### Query와 데이터 증가

Projection과 Batch Loader는 잘 활용되지만 무제한 목록, Count Query, 복잡한 동적 필터는 데이터가 늘면 비용이 커집니다. 주요 Endpoint의 Query 수·P95·Heap을 Baseline으로 만들고 Cursor, Export Job, 검색 Field Allowlist로 통제해야 합니다.

### 부하 테스트 운영화

k6에 Smoke·Baseline·Load·Stress·Spike와 실제 업무 Scenario가 준비되어 있다는 점은 강점입니다. 다만 VU 기반 Closed Model과 전역 Threshold 중심이라 일정 RPS 유입, API별 SLO, 과부하 뒤 회복 시간을 충분히 판정하지 못합니다. Credential 기본값을 제거하고 Arrival-rate Scenario, Journey별 Threshold, Test Data Cleanup, Prometheus·Trace 자동 연결을 추가해야 합니다. 자세한 기준은 [[k6 부하 테스트와 성능 검증]]에 정리합니다.

### 관측성 표준화

Observation과 Structured Log가 적용됐지만 Service·Environment·Error Tag의 이름, Cardinality Budget, Signal 손실 기준이 더 필요합니다. 요청 하나가 Metric·Log·Trace에서 같은 식별자로 연결되고 Collector 장애 시 Drop 수를 알아야 운영화 수준입니다.

### CI/CD 복구성

Pipeline 단계는 풍부하지만 Stateful Jenkins Controller와 PVC는 단일 실패 지점입니다. JCasC, Ephemeral Agent, PVC Snapshot Restore, Webhook 전용 공개 경로를 검증해야 합니다.

### 비용 관리

Kubecost를 설치하는 것만으로 팀별 비용이 정확해지지 않습니다. Team·Product·Environment Label을 강제하고 Shared Cost 배분 규칙과 Right-sizing 승인 절차를 정해야 합니다.

## 10. 최종 완료 기준

- [ ] 저장소와 CI Artifact에 평문 Secret이 없습니다.
- [ ] AZ, Node, DB, Redis, Collector 장애 훈련이 RTO/RPO를 만족합니다.
- [ ] 환경 간 Network와 Resource 경계가 자동 Test로 검증됩니다.
- [ ] 핵심 User Journey에 SLI, SLO, Alert, Runbook이 연결됩니다.
- [ ] 비동기 Side Effect가 중복·유실 없이 최종 상태로 수렴합니다.
- [ ] 주요 Query와 Workload가 목표 데이터·부하에서 P95 기준을 만족합니다.
- [ ] Architecture 예외에는 Owner와 만료일이 있고 신규 위반은 CI가 차단합니다.
- [ ] 검증한 동일 Image Digest가 환경을 거쳐 승격되고 즉시 Rollback 가능합니다.

이 기준을 만족해야 기술을 단순히 도입한 상태가 아니라 운영 가능한 시스템으로 만들었다고 볼 수 있습니다.
