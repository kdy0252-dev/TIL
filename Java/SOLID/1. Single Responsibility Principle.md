# Single Responsibility Principle

SRP는 “Class는 Method 하나만 가져야 한다”가 아니라 **하나의 Actor 또는 변경 이유를 가져야 한다**는 원칙이다. 업무 규칙, 영속성 Mapping, HTTP 계약과 외부 Message Format은 서로 다른 이유로 바뀌므로 같은 Class에 섞지 않는다.

## 위반 예제

```java
@Service
@RequiredArgsConstructor
public class BookingService {

    private final BookingJpaRepository repository;
    private final ObjectMapper objectMapper;
    private final MailClient mailClient;

    @Transactional
    public BookingResponse cancel(long bookingId, String reason) {
        BookingJpaEntity entity = repository.findById(bookingId)
                                            .orElseThrow();
        entity.setStatus("CANCELLED");
        entity.setCancellationReason(reason);
        repository.save(entity);

        String payload = Try.of(() -> objectMapper.writeValueAsString(entity))
                            .getOrElseThrow(cause -> new IllegalStateException(cause));
        mailClient.send(entity.getPassengerEmail(), payload);
        return new BookingResponse(entity.getId(), entity.getStatus());
    }
}
```

이 Class는 취소 가능 여부, JPA State, JSON, Email과 Web Response까지 책임진다. 하나의 변경이 다른 책임의 Test와 Transaction에 영향을 준다.

## 책임을 계층별로 분리하기

### Domain Model: 업무 규칙

```java
@Getter
@Builder(access = AccessLevel.PRIVATE)
@AllArgsConstructor(access = AccessLevel.PRIVATE)
public final class Booking {

    private final long id;
    private final BookingStatus status;
    private final Cancellation cancellation;
    private final Audit audit;

    public Either<BookingError, Booking> cancel(
        CancellationReason reason,
        AuditActor actor,
        Instant cancelledAt
    ) {
        return Either.cond(
            status.canCancel(),
            () -> Booking.builder()
                         .id(id)
                         .status(BookingStatus.CANCELLED)
                         .cancellation(new Cancellation(reason, cancelledAt))
                         .audit(audit.updatedBy(actor))
                         .build(),
            () -> new BookingError.InvalidCancellationState(id, status)
        );
    }
}
```

### Port: Application이 필요한 계약

```java
public interface BookingPort {
    Either<BookingError, Booking> findById(long bookingId);
    Either<BookingError, Booking> save(Booking booking);
}

public interface BookingEventOutboxPort {
    Either<BookingError, Void> append(BookingCancelled event);
}
```

### Application Service: 업무 흐름 조율

```java
@Service
@RequiredArgsConstructor
public class CancelBookingService implements CancelBookingUseCase {

    private final BookingPort bookingPort;
    private final BookingEventOutboxPort outboxPort;
    private final Clock clock;
    private final BookingExceptionMapper exceptionMapper;

    @Override
    @Transactional
    public BookingResource cancel(CancelBookingCommand command) {
        return bookingPort.findById(command.bookingId())
                          .flatMap(booking -> booking.cancel(
                              command.reason(),
                              command.actor(),
                              clock.instant()
                          ))
                          .flatMap(bookingPort::save)
                          .flatMap(saved -> outboxPort.append(BookingCancelled.from(saved))
                                                      .map(ignored -> saved))
                          .map(BookingResource::from)
                          .getOrElseThrow(exceptionMapper::toException);
    }
}
```

### Adapter: 기술 세부사항

Persistence Adapter는 JPA Entity Mapping과 Database 예외 변환만 맡고, Notification Consumer는 Commit된 Event를 Email Provider 계약으로 변환한다. Email 실패 때문에 취소 Transaction이 모호해지지 않도록 Outbox를 사용한다.

```java
@Component
@RequiredArgsConstructor
public class BookingPersistenceAdapter implements BookingPort {

    private final BookingJpaRepository repository;
    private final BookingPersistenceMapper mapper;

    @Override
    public Either<BookingError, Booking> findById(long bookingId) {
        return repository.findById(bookingId)
                         .map(mapper::toDomain)
                         .<Either<BookingError, Booking>>map(Either::right)
                         .orElseGet(() -> Either.left(new BookingError.NotFound(bookingId)));
    }

    @Override
    public Either<BookingError, Booking> save(Booking booking) {
        return Try.of(() -> repository.save(mapper.toEntity(booking)))
                  .map(mapper::toDomain)
                  .toEither()
                  .mapLeft(cause -> new BookingError.PersistenceFailure(booking.getId(), cause));
    }
}
```

## 분리 기준

- 서로 다른 사람이 서로 다른 이유로 변경하는가?
- Transaction과 실패 정책이 다른가?
- 단위 Test 입력과 출력이 다른가?
- Domain이 Framework Annotation과 외부 DTO를 알아야 하는가?
- Class 이름을 설명할 때 “그리고”가 반복되는가?

작은 Method를 무조건 별도 Class로 나누면 탐색 비용만 늘어난다. 하나의 응집된 업무 규칙은 Domain Model 안에 함께 있어야 한다.

## 기억할 점

SRP는 물리적인 File 수가 아니라 변경의 응집도를 다룬다. Domain은 업무 규칙, Service는 흐름, Port는 필요한 계약, Adapter는 기술 변환을 맡게 하면 변경 이유와 Test 경계가 자연스럽게 분리된다.
