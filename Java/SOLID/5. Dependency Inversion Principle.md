# Dependency Inversion Principle

DIP는 고수준 업무 정책이 Database, HTTP SDK와 Message Broker 같은 저수준 세부사항에 직접 의존하지 않게 하는 원칙이다. 둘 다 Application이 정의한 추상 계약에 의존하고, Adapter가 그 계약을 구현한다.

## 직접 의존의 문제

```java
@Service
@RequiredArgsConstructor
public class CompletePaymentService {

    private final StripeClient stripeClient;
    private final PaymentJpaRepository repository;

    public PaymentResponse complete(CompletePaymentCommand command) {
        StripeCharge charge = stripeClient.charge(command.token(), command.amount());
        PaymentJpaEntity saved = repository.save(PaymentJpaEntity.from(charge));
        return PaymentResponse.from(saved);
    }
}
```

Use Case가 Provider DTO, JPA Entity와 예외 계약에 결합된다. 결제 Provider 변경이 업무 Service를 수정하게 하고 Test도 실제 SDK 형태를 알아야 한다.

## Application이 Port를 소유한다

```java
public interface ApprovePaymentPort {
    Either<PaymentError, PaymentApproval> approve(PaymentRequest request);
}

public interface SavePaymentPort {
    Either<PaymentError, Payment> save(Payment payment);
}
```

Port는 외부 SDK 기능을 그대로 복사하지 않고 Use Case가 필요한 업무 언어를 사용한다.

```java
@Service
@RequiredArgsConstructor
public class CompletePaymentService implements CompletePaymentUseCase {

    private final ApprovePaymentPort approvePaymentPort;
    private final SavePaymentPort savePaymentPort;

    @Override
    public Either<PaymentError, PaymentResource> complete(CompletePaymentCommand command) {
        return Payment.create(command.bookingId(), command.amount(), command.actor())
                      .flatMap(payment -> approvePaymentPort.approve(payment.toRequest())
                                                          .flatMap(payment::approve))
                      .flatMap(savePaymentPort::save)
                      .map(PaymentResource::from);
    }
}
```

## Adapter가 기술을 변환한다

```java
@Component
@RequiredArgsConstructor
public class StripePaymentAdapter implements ApprovePaymentPort {

    private final StripeClient stripeClient;
    private final StripePaymentMapper mapper;

    @Override
    public Either<PaymentError, PaymentApproval> approve(PaymentRequest request) {
        return Try.of(() -> stripeClient.charge(mapper.toStripeRequest(request)))
                  .map(mapper::toApproval)
                  .toEither()
                  .mapLeft(cause -> mapFailure(request, cause));
    }

    private PaymentError mapFailure(PaymentRequest request, Throwable cause) {
        return cause instanceof StripeDeclinedException declined
            ? new PaymentError.Declined(request.paymentId(), declined.getDeclineCode())
            : new PaymentError.ProviderUnavailable(request.paymentId(), cause);
    }
}
```

Provider Error Code와 DTO는 Adapter 밖으로 나가지 않는다. Service는 `Declined`와 `ProviderUnavailable`이라는 업무 의미만 처리한다.

## 의존 방향

```text
Web Adapter -> In Port -> Application Service -> Domain
                               |
                               v
                            Out Port
                               ^
                               |
                    Payment/Persistence Adapter
```

Source Code Dependency가 안쪽을 향해야 한다. Runtime에는 Spring이 Adapter 구현을 Service에 주입하지만 Compile-time에는 Adapter가 Application Port를 참조한다.

## Dependency Injection과의 차이

Constructor Injection은 객체를 연결하는 기술이다. 구체 `StripeClient`를 Constructor로 주입해도 여전히 저수준 세부사항에 의존하므로 DIP를 만족하지 않는다. 추상화의 소유권과 의존 방향이 핵심이다.

## Test

```java
@Test
void completesPaymentAfterProviderApproval() {
    ApprovePaymentPort approvePaymentPort = request -> Either.right(
        new PaymentApproval(request.paymentId(), "provider-transaction-id")
    );
    SavePaymentPort savePaymentPort = Either::right;
    CompletePaymentService service = new CompletePaymentService(approvePaymentPort, savePaymentPort);

    Either<PaymentError, PaymentResource> result = service.complete(fixture.command());

    assertThat(result.get().status()).isEqualTo(PaymentStatus.APPROVED);
}
```

Test만을 위해 Production Interface를 만들지는 않는다. 외부 변경 가능성, 계층 경계와 업무 언어가 있을 때 Port를 만든다. 단순 순수 계산은 구체 Class를 직접 주입해도 된다.

## 기억할 점

DIP는 “Interface를 많이 만드는 원칙”이 아니다. 업무 정책이 기술 Vendor와 Framework보다 오래 살아남도록 추상 계약의 소유권을 Application 쪽으로 뒤집는 원칙이다. Adapter는 외부 DTO·예외를 Domain 언어로 변환하고 Service는 Port를 통해 업무 흐름만 표현한다.
