# Dependency Inversion Principle
Dependency Inversion Principle(DIP)은 SOLID 원칙 중 하나로, 다음 두 가지 규칙을 따른다.

1.  고차원 모듈은 저차원 모듈에 의존해서는 안 된다. 둘 모두 추상화에 의존해야 한다.
2.  추상화는 세부 사항에 의존해서는 안 된다. 세부 사항이 추상화에 의존해야 한다.

- 구체적인 클래스보다 인터페이스나 추상 클래스에 의존해야 한다
- 고수준 모듈이 저수준 모듈의 구현에 의존하지 않도록 한다

## 고수준과 저수준은 무엇인가

“주문을 확정하고 결제를 요청한다”는 업무 정책은 고수준이다. HTTP Client로 특정 결제사의 URL을 호출하는 것은 저수준 세부사항이다. 업무 정책이 SDK Class와 JSON 형식을 직접 알면 Provider 교체와 Test가 모두 Domain Code의 수정이 된다.

```text
잘못된 방향
OrderService -> SpecificPaymentSdk

DIP 적용
OrderService -> PaymentPort <- PaymentSdkAdapter
```

화살표는 Source Code Dependency다. Runtime 호출은 Service에서 Adapter로 진행되지만 Compile-time 의존성은 Adapter가 업무가 정의한 Port를 구현하는 방향으로 뒤집힌다.
## DIP가 필요한 이유
- **코드의 유연성 향상**: DIP를 준수하면 고차원 모듈이 저차원 모듈의 구현에 의존하지 않으므로, 코드의 유연성이 향상된다.
- **재사용성 향상**: DIP를 준수하면 모듈 간의 결합도가 낮아지므로, 코드의 재사용성이 높아진다.
- **유지보수성 향상**: DIP를 준수하면 모듈 간의 의존성이 줄어들므로, 유지보수성이 향상된다.
## 예시
다음과 같은 `Button` 클래스와 `LightBulb` 클래스가 있다고 가정해 보자.

```java
public class Button {
    private LightBulb lightBulb;

    public Button(LightBulb lightBulb) {
        this.lightBulb = lightBulb;
    }

    public void press() {
        lightBulb.turnOn();
    }
}

public class LightBulb {
    public void turnOn() {
        System.out.println("LightBulb is on");
    }
}
```

위 예시에서 `Button` 클래스는 `LightBulb` 클래스에 직접 의존한다. 즉, `Button` 클래스는 `LightBulb` 클래스의 구현에 의존한다. 만약 `LightBulb` 클래스를 다른 클래스로 변경해야 한다면, `Button` 클래스도 함께 변경해야 한다.

이러한 문제를 해결하기 위해서는 `Button` 클래스가 `LightBulb` 클래스에 직접 의존하는 것이 아니라, 추상화된 인터페이스에 의존하도록 변경해야 한다.

```java
public interface Switchable {
    void turnOn();
}

public class LightBulb implements Switchable {
    @Override
    public void turnOn() {
        System.out.println("LightBulb is on");
    }
}

public class Button {
    private Switchable switchable;

    public Button(Switchable switchable) {
        this.switchable = switchable;
    }

    public void press() {
        switchable.turnOn();
    }
}
```

이제 `Button` 클래스는 `Switchable` 인터페이스에 의존한다. 즉, `Button` 클래스는 `LightBulb` 클래스의 구현에 의존하지 않는다. 만약 `LightBulb` 클래스를 다른 클래스로 변경해야 한다면, `Button` 클래스는 변경할 필요가 없다.

## Interface는 누가 소유해야 하는가

저수준 SDK를 그대로 닮은 Interface를 Application에 두면 이름만 추상화됐을 뿐 의존성은 뒤집히지 않는다.

```java
// 업무가 필요한 계약
interface PaymentPort {
    PaymentResult pay(PaymentCommand command);
}

// 외부 SDK 세부사항을 변환하는 Adapter
final class ExternalPaymentAdapter implements PaymentPort {
    private final ExternalPaymentClient client;

    public PaymentResult pay(PaymentCommand command) {
        ExternalRequest request = map(command);
        return map(client.approve(request));
    }
}
```

Port의 Method와 Type은 Consumer인 Application의 언어로 정의한다. 외부 SDK Request·Response가 Port를 통과하지 않게 한다.

## Dependency Injection과 DIP의 차이

Constructor Injection은 객체를 밖에서 전달하는 구현 기법이다. 구체 Class를 주입해도 DI는 사용한 것이지만 DIP를 만족한다고 할 수는 없다.

```java
OrderService(ExternalPaymentClient client) // DI지만 세부사항에 직접 의존
OrderService(PaymentPort paymentPort)       // DI를 이용해 DIP 구현
```

Spring Container는 Wiring을 도울 뿐 올바른 추상화 방향을 자동으로 만들어주지 않는다.

## Hexagonal Architecture와 DIP

Hexagonal Architecture의 Port와 Adapter가 DIP를 구조적으로 적용한다.

- In Port는 외부가 Application에 요청할 Use Case 계약이다.
- Out Port는 Application이 Database·Message·외부 API에 요구하는 계약이다.
- Adapter는 Framework와 Protocol을 업무 Type으로 변환한다.
- Domain은 Spring, JPA와 HTTP를 몰라도 된다.

Architecture Test로 Domain이 Adapter Package를 참조하지 않는지 검사하면 원칙을 실행 가능한 규칙으로 바꿀 수 있다.
## 장점
- 코드의 유연성 향상
- 재사용성 향상
- 유지보수성 향상
## 주의사항
- 추상화에 너무 의존하면 코드가 장황해지고, 클래스 간의 관계를 파악하기 어려워질 수 있다.
- DIP를 적용할 때는 변경될 가능성이 있는 부분에 대해서만 추상화를 적용하는 것이 좋다.

## Test Double만을 위한 Interface

Test에서 Mock하기 쉽다는 이유만으로 모든 Class 앞에 Interface를 두면 Code 탐색 비용이 늘어난다. 외부 Side Effect, 교체 가능한 구현 또는 중요한 Architecture Boundary인지 먼저 판단한다. 순수 계산 Class는 실제 객체로 Test해도 충분할 수 있다.

## 기억할 점

DIP의 핵심은 “Interface에 의존한다”가 아니라 **업무 정책이 자신에게 필요한 추상화를 소유하고 기술 세부사항이 그 계약에 맞춰 들어오게 하는 것**이다. 의존성 방향이 바뀌지 않았다면 이름만 Port로 바꿔도 효과가 없다.
