# Interface Segregation Principle

ISP는 Client가 사용하지 않는 Method에 의존하지 않게 Interface를 역할별로 나누는 원칙이다. 구현 Class가 편하다는 이유로 CRUD 전체를 하나의 Interface에 넣으면 조회 Client도 삭제·수정 계약에 결합된다.

## 비대한 Port

```java
public interface BookingPort {
    Either<BookingError, Booking> findById(long bookingId);
    Either<BookingError, Page<Booking>> search(BookingSearchCondition condition);
    Either<BookingError, Booking> save(Booking booking);
    Either<BookingError, Void> delete(long bookingId);
    Either<BookingError, byte[]> exportExcel(BookingSearchCondition condition);
}
```

Persistence, 검색, 삭제와 Excel이라는 서로 다른 Client 요구가 섞였다. In-memory Test Double과 Read Replica Adapter도 불필요한 Method를 구현해야 한다.

## Client 기준으로 분리하기

```java
public interface LoadBookingPort {
    Either<BookingError, Booking> findById(long bookingId);
}

public interface SearchBookingPort {
    Either<BookingError, Page<BookingProjection>> search(BookingSearchCondition condition);
}

public interface SaveBookingPort {
    Either<BookingError, Booking> save(Booking booking);
}

public interface ExportBookingPort {
    Either<BookingExportError, ExportedFile> export(BookingSearchCondition condition);
}
```

하나의 Adapter가 여러 Port를 구현하는 것은 괜찮다. Interface 분리는 구현 Class 수가 아니라 Client 의존성을 줄이는 일이다.

```java
@Component
@RequiredArgsConstructor
public class BookingPersistenceAdapter
    implements LoadBookingPort, SearchBookingPort, SaveBookingPort {

    private final BookingJpaRepository repository;
    private final BookingPersistenceMapper mapper;

    @Override
    public Either<BookingError, Booking> findById(long bookingId) {
        return repository.findById(bookingId)
                         .map(mapper::toDomain)
                         .<Either<BookingError, Booking>>map(Either::right)
                         .orElseGet(() -> Either.left(new BookingError.NotFound(bookingId)));
    }

    @Override
    public Either<BookingError, Booking> save(Booking booking) {
        return Try.of(() -> repository.save(mapper.toEntity(booking)))
                  .map(mapper::toDomain)
                  .toEither()
                  .mapLeft(BookingError.PersistenceFailure::new);
    }
}
```

## In Port도 Use Case별로 분리한다

```java
public interface GetBookingUseCase {
    Either<BookingError, BookingResource> get(long bookingId);
}

public interface CancelBookingUseCase {
    Either<BookingError, BookingResource> cancel(CancelBookingCommand command);
}

public interface SearchBookingUseCase {
    Either<BookingError, Page<BookingListResource>> search(SearchBookingCommand command);
}
```

Controller는 필요한 Use Case만 주입받는다. 다른 Vertical Slice가 공개 계약을 사용할 때도 거대한 Service Interface 전체에 의존하지 않는다.

## 너무 잘게 나누지 않기

항상 Method 하나당 Interface 하나가 정답은 아니다. 같은 Client가 항상 함께 사용하고 같은 변경 이유·권한·Transaction 성격을 가진 Method는 한 역할 Interface에 둘 수 있다.

다음 신호가 있을 때 분리를 검토한다.

- 구현체가 `UnsupportedOperationException`을 던진다.
- Adapter의 Method 절반이 빈 구현이다.
- Test Double이 Test와 관계없는 Method를 계속 구현한다.
- 조회 Consumer가 쓰기 Dependency까지 가져간다.
- Interface의 Error 타입이 Method마다 서로 다른 의미를 억지로 합친다.

## DIP와의 차이

ISP는 Interface의 크기와 Client별 역할을 다루고, DIP는 고수준 정책이 저수준 구현에 직접 의존하지 않게 한다. 작은 Interface라도 Adapter Package가 소유해 Application이 가져다 쓰면 의존 방향은 여전히 잘못될 수 있다.

## 기억할 점

좋은 Interface는 구현체의 모든 기능 목록이 아니라 Client가 수행할 역할의 최소 계약이다. Port와 Use Case를 Client 관점에서 나누면 변경 영향, Test Double과 권한 경계가 함께 작아진다.
