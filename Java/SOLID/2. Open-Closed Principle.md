# Open-Closed Principle
Open-Closed Principle(OCP)은 SOLID 원칙 중 하나로, 소프트웨어 개체(클래스, 모듈, 함수 등)는 확장에 열려 있어야 하고, 수정에는 닫혀 있어야 한다는 원칙이다. 즉, 기존 코드를 수정하지 않고 기능을 추가할 수 있도록 설계해야 한다.

여기서 “수정에 닫힌다”는 기존 Code를 영원히 수정하지 않는다는 뜻이 아니다. **이미 안정화된 핵심 흐름을 매번 고치지 않고, 예상한 변경 축을 새로운 구현 추가로 수용한다**는 의미다.

## 먼저 변경 축을 찾는다

할인 정책이 늘어날 때마다 다음 Method를 수정한다고 가정한다.

```java
Money discount(Order order, DiscountType type) {
    if (type == FIXED) {
        return fixedDiscount(order);
    }
    if (type == RATE) {
        return rateDiscount(order);
    }
    throw new IllegalArgumentException("unsupported type");
}
```

새 정책이 추가될 때 기존 조건문을 열어 수정하고 모든 분기를 다시 Test해야 한다. 변화가 반복되는 축이 “할인 계산 방법”임을 알았다면 이를 추상화한다.

```java
interface DiscountPolicy {
    boolean supports(Order order);
    Money calculate(Order order);
}

final class RateDiscountPolicy implements DiscountPolicy {
    public boolean supports(Order order) {
        return order.member().isVip();
    }

    public Money calculate(Order order) {
        return order.totalPrice().multiply(0.1);
    }
}
```

Orchestrator는 구현 목록에서 적용 가능한 정책을 선택한다.

```java
final class DiscountService {
    private final List<DiscountPolicy> policies;

    Money discount(Order order) {
        return policies.stream()
            .filter(policy -> policy.supports(order))
            .findFirst()
            .map(policy -> policy.calculate(order))
            .orElse(Money.ZERO);
    }
}
```

새 정책은 새 Class와 등록으로 추가할 수 있다. 핵심 선택 흐름은 변하지 않는다.

- **확장에 열려 있음**: 새로운 요구사항이나 변경사항에 따라 기능을 확장할 수 있어야 한다.
- **수정에 닫혀 있음**: 기존 코드를 수정하지 않고 확장할 수 있도록 설계되어야 한다.

OCP를 준수하면 코드의 안정성을 유지하면서 유연하게 변화에 대응할 수 있다.
## OCP가 필요한 이유
- **유지보수성 향상**: 코드를 수정하지 않고 기능을 추가할 수 있으므로, 버그 발생 가능성이 줄어들고 유지보수가 용이해진다.
- **재사용성 향상**: 기존 코드를 수정하지 않고 기능을 확장할 수 있으므로, 코드 재사용성이 높아진다.
- **안정성 향상**: 기존 코드를 수정하지 않으므로, 시스템의 안정성이 향상된다.
## 디자인 패턴 예시: 전략 패턴
전략 패턴은 OCP를 잘 보여주는 디자인 패턴 중 하나다. 전략 패턴을 사용하면 알고리즘군을 정의하고 각각을 캡슐화하여 필요에 따라 교체할 수 있다.

```java
public interface Strategy {
    int execute(int a, int b);
}
```

```java
public class AddStrategy implements Strategy {
    public int execute(int a, int b) {
        return a + b;
    }
}
```

```java
public class SubtractStrategy implements Strategy {
    public int execute(int a, int b) {
        return a - b;
    }
}
```

```java
public class Calculator {
    private Strategy strategy;

    public Calculator(Strategy strategy) {
        this.strategy = strategy;
    }

    public int calculate(int a, int b) {
        return strategy.execute(a, b);
    }
}
```

위 예시에서 `계산기` 클래스는 `전략` 인터페이스에 의존한다. 새로운 연산이 필요할 경우, `전략` 인터페이스를 구현하는 새로운 클래스를 추가하기만 하면 된다. `계산기` 클래스의 코드는 수정할 필요가 없다.

## 장점
- 코드 변경 없이 새로운 기능 추가 가능
- 기존 기능에 영향 없이 확장 가능
- 유지보수 용이
- 코드 재사용성 증가
## 주의사항
- 모든 확장에 대해 코드를 열어두면 유지보수가 어려워질 수 있다. 적절한 추상화 수준을 유지해야 한다.
- 과도한 추상화는 코드 복잡성을 증가시킬 수 있으므로, 필요한 경우에만 OCP를 적용해야 한다.

## OCP는 모든 변화에 동시에 닫힐 수 없다

Policy 선택 우선순위가 바뀌거나 여러 할인을 중첩해야 한다면 `DiscountService` 자체는 수정될 수 있다. 설계는 현재 알고 있는 변경 축에 대해서만 닫혀 있다. 미래의 모든 요구를 예측해 추상화를 만들면 사용되지 않는 Interface와 Factory만 늘어난다.

다음 신호가 반복될 때 확장을 검토한다.

- 같은 `if/switch`에 Variant가 계속 추가된다.
- 결제 수단, 알림 Channel, 정렬 전략처럼 Algorithm이 교체된다.
- 외부 Provider마다 Integration 구현이 다르다.
- Feature별 배포 주기와 소유 팀이 다르다.

한 번만 존재하는 구현에 Interface를 추가하는 것이 항상 OCP는 아니다. 변경 이유가 실제로 확인됐는지 먼저 본다.

## Data-driven 설계와 비교

변경이 Code가 아니라 값의 추가라면 Polymorphism보다 설정 Table이 단순할 수 있다.

```text
배송비 정책: 국가와 무게별 숫자 Table
복잡한 결제 처리: Provider별 Strategy
```

규칙이 Data로 표현 가능한데 Class를 계속 만들면 배포 없이 바꿀 수 있는 값까지 Code 변경이 된다. 반대로 복잡한 업무 행위를 설정식 하나에 넣으면 Type Safety와 Test가 약해진다.

## Test 관점

OCP의 효과는 기존 Test를 거의 건드리지 않고 새 구현의 Contract Test만 추가할 수 있을 때 드러난다. 모든 구현이 같은 불변식을 지켜야 한다면 Interface Contract Test를 재사용한다. 등록 누락과 정책 우선순위는 통합 Test로 별도 검증한다.

## 기억할 점

OCP의 목적은 Interface 개수를 늘리는 것이 아니다. 자주 변하는 부분을 안정적인 핵심 흐름 밖으로 밀어내 변화의 영향 범위를 줄이는 것이다. 실제 변경 이력을 보고 추상화해야 한다.
