# Java에서의 함수형 프로그래밍

함수형 프로그래밍은 데이터를 여러 단계의 함수로 변환하고, 상태 변경과 Side Effect를 명시적인 경계로 밀어내는 방식이다. Java에서는 순수 함수, 불변 Value Object, Stream, `Optional`, Vavr `Try`·`Either`·`Validation`을 조합해 적용할 수 있다.

함수형 스타일의 목적은 `for`와 `if`를 기계적으로 없애는 것이 아니다. 입력과 출력의 관계를 드러내고, 실패·상태 변경·외부 I/O가 섞이지 않게 만드는 것이다.

## 핵심 원칙

### 순수 함수

순수 함수는 같은 입력에 같은 출력을 반환하고 외부 상태를 변경하지 않는다. 시간, Random, Database와 전역 변수에 직접 의존하면 순수하지 않다.

```java
public record Fare(long baseFare, long distanceFare, long extraCharge) {

    public Fare {
        if (baseFare < 0 || distanceFare < 0 || extraCharge < 0) {
            throw new IllegalArgumentException("Fare components must not be negative");
        }
    }

    public long total() {
        return Math.addExact(Math.addExact(baseFare, distanceFare), extraCharge);
    }
}

public final class DiscountPolicy {

    public long apply(Fare fare, Discount discount) {
        long discountAmount = discount.calculate(fare.total());
        return Math.max(0L, Math.subtractExact(fare.total(), discountAmount));
    }
}
```

`apply`는 외부 값을 읽거나 바꾸지 않으므로 입력만으로 단위 Test할 수 있다. 현재 시각이 필요하면 `Instant.now()`를 내부에서 호출하지 않고 `Clock` 또는 계산 기준 시각을 인자로 받는다.

### 불변성

`final List`는 참조만 바꿀 수 없게 할 뿐 내부 원소 변경을 막지 않는다. 생성 시 방어적 복사를 하고 상태 변경은 새 값을 반환한다.

```java
public record Route(List<RoutePoint> points) {

    public Route {
        points = List.copyOf(points);
    }

    public Route append(RoutePoint point) {
        List<RoutePoint> appended = Stream.concat(points.stream(), Stream.of(point))
                                                .toList();
        return new Route(appended);
    }
}
```

큰 Collection을 매번 복사하면 비용이 크다. Hot Path에서는 Immutable Snapshot의 수명, Batch Update 또는 전용 Persistent Collection을 Profile 결과에 따라 선택한다.

### 고차 함수와 정책 주입

함수를 인자로 받거나 반환하는 함수를 고차 함수라 한다. 단순 숫자 예제보다 업무 정책을 조합할 때 가치가 크다.

```java
@FunctionalInterface
public interface FareRule {
    FareBreakdown apply(FareContext context, FareBreakdown current);
}

public final class FareCalculator {

    private final List<FareRule> rules;

    public FareCalculator(List<FareRule> rules) {
        this.rules = List.copyOf(rules);
    }

    public FareBreakdown calculate(FareContext context) {
        return rules.stream()
                    .reduce(
                        FareBreakdown.empty(),
                        (current, rule) -> rule.apply(context, current),
                        (left, right) -> {
                            throw new UnsupportedOperationException("Fare rules must be evaluated sequentially");
                        }
                    );
    }
}
```

Rule 순서가 결과에 영향을 주므로 `parallelStream()`을 사용하지 않는다. 결합 법칙이 성립하지 않는 Reduction을 병렬화하면 결과가 달라진다.

## Collection 변환을 Pipeline으로 표현하기

명령형 Code는 임시 Collection과 분기를 숨겨 놓기 쉽다.

```java
List<SettlementLine> result = new ArrayList<>();
for (Booking booking : bookings) {
    if (booking.isCompleted() && booking.getFare() != null) {
        result.add(SettlementLine.from(booking));
    }
}
```

Stream은 “완료된 예약 선택 -> 정산 가능 운임 추출 -> 정산 Line 변환”이라는 단계가 드러난다.

```java
List<SettlementLine> settlementLines = bookings.stream()
                                                .filter(Booking::isCompleted)
                                                .map(Booking::fare)
                                                .flatMap(Optional::stream)
                                                .map(SettlementLine::from)
                                                .toList();
```

Lambda Parameter는 `b`, `x`, `it`보다 `booking`, `fare`처럼 의미를 드러낸다. Lambda가 Method 호출 하나뿐이면 Method Reference를 우선한다.

## 실패도 Pipeline에 포함하기

외부 API 단계는 `Try`, 예상 가능한 업무 실패는 `Either`, 여러 독립 검증은 `Validation`으로 표현한다.

```java
public DispatchPlan createPlan(CreateDispatchCommand command) {
    return validate(command)
        .toEither()
        .mapLeft(errors -> new DispatchError.InvalidCommand(errors.toJavaList()))
        .flatMap(validCommand -> passengerPort.findAllByIds(validCommand.passengerIds()))
        .flatMap(passengers -> vehiclePort.findAvailable(validCommand.serviceDate())
                                           .map(vehicles -> new DispatchCandidates(passengers, vehicles)))
        .map(dispatchOptimizer::optimize)
        .flatMap(DispatchPlan::validateFeasibility)
        .getOrElseThrow(dispatchExceptionMapper::toException);
}
```

Service는 업무 흐름만 열거한다. Command Field 검증, 후보 생성과 최적화 세부 구현은 각각의 Domain Component에 둔다.

## Side Effect는 가장자리로 모은다

Database 저장, HTTP 호출, Message 발행과 Log는 Side Effect다. 계산 중간에 섞으면 같은 입력을 재실행하기 어렵다.

```java
public SettlementResource settle(SettleFareCommand command) {
    return settlementPort.findByBookingId(command.bookingId())
                         .flatMap(settlement -> settlement.complete(command.payment()))
                         .flatMap(settlementPort::save)
                         .flatMap(saved -> outboxPort.append(SettlementCompleted.from(saved))
                                                       .map(ignored -> saved))
                         .map(SettlementResource::from)
                         .getOrElseThrow(settlementExceptionMapper::toException);
}
```

Event 발행이 필수라면 단순 `peek`에 넣지 않는다. Outbox 저장처럼 실패가 반환값에 포함되는 단계로 만든다.

## Optional을 분기문 대신 사용하기

```java
public Optional<String> resolvePrimaryPhone(Passenger passenger) {
    return Optional.ofNullable(passenger.contact())
                   .map(Contact::phoneNumbers)
                   .stream()
                   .flatMap(List::stream)
                   .filter(PhoneNumber::primary)
                   .map(PhoneNumber::value)
                   .findFirst();
}
```

값이 없다는 사실이 업무 실패라면 Out Port나 Domain 내부에서 `Either<PassengerError, String>`으로 명시한다. 최상위 Service는 이를 Application Exception으로 변환한다.

## Validation으로 Error 누적하기

`Either`는 첫 실패에서 멈춘다. 사용자 입력 Field 여러 개의 Error를 한 번에 보여 주려면 `Validation.combine`이 적합하다.

```java
public Validation<Seq<ValidationError>, CreatePassengerCommand> validate(
    CreatePassengerRequest request
) {
    return Validation.combine(
                         ValidationUtil.notBlank(request.name(), "Name is required"),
                         ValidationUtil.validPhone(request.phone(), "Phone is invalid"),
                         ValidationUtil.notNull(request.serviceType(), "Service type is required")
                     )
                     .ap((name, phone, serviceType) ->
                         new CreatePassengerCommand(name, phone, serviceType));
}
```

## Parallel Stream을 기본값으로 쓰지 않는다

`parallelStream()`은 Thread 수만 늘리면 빨라지는 기능이 아니다. 작업 분할 비용, Common ForkJoinPool 공유, Encounter Order, Blocking I/O와 Reduction 결합 법칙을 검토해야 한다.

- CPU 연산이 충분히 크고 서로 독립적인가?
- 입력 크기가 병렬화 비용보다 큰가?
- Collector와 Reduction이 Associative한가?
- Common Pool을 다른 기능과 공유해도 되는가?
- 부하 Test에서 P95/P99가 실제로 개선되는가?

Web 요청 내부의 Blocking I/O를 `parallelStream()`으로 호출하면 Timeout과 Thread 고갈을 더 악화시킬 수 있다. 명시적인 Executor, Virtual Thread 또는 비동기 Client를 목적에 맞게 사용한다.

## 함수형 스타일이 부적절한 경우

- 단순 Guard Clause가 긴 `Optional` Chain보다 명확한 경우
- 성능이 검증된 Tight Loop에서 Stream 할당이 실제 병목인 경우
- 상태 Machine처럼 전이가 핵심인데 Chain이 상태를 숨기는 경우
- Debugging을 위해 명시적인 단계와 이름이 필요한 경우

함수형 문법이 아니라 Referential Transparency, 불변성, 합성 가능성과 Side Effect 격리가 목표다.

## 검토 Checklist

- 입력 Collection을 수정하지 않고 새 값을 반환하는가?
- `for` + `if` + `add`를 `filter`·`map`·`toList`로 표현할 수 있는가?
- `null`과 예외가 Method Signature에 숨겨져 있지 않은가?
- Lambda가 길어지면 이름 있는 Method나 Domain Service로 분리했는가?
- `peek`와 `forEach`에서 필수 상태 변경을 하지 않는가?
- `parallelStream()`의 결합 법칙과 부하 결과를 확인했는가?
- Stream Pipeline 안에서 Database를 N번 호출하지 않는가?

## 기억할 점

좋은 함수형 Java Code는 문법적으로 짧은 Code가 아니다. Domain Value는 불변이고, 각 함수는 한 변환을 설명하며, 실패는 타입으로 흐르고, Side Effect는 Port 경계에 모여 있어야 한다.
