# 반응형 프로그래밍

반응형 프로그래밍은 값 하나를 즉시 반환하는 대신 시간이 지나며 도착하는 0개 이상의 값과 완료·실패 Signal을 Publisher로 표현한다. Spring WebFlux와 Reactor에서는 `Mono<T>`가 0~1개, `Flux<T>`가 0개 이상의 값을 나타낸다.

## 비동기와 반응형은 다르다

`@Async Future<T>`는 작업 하나의 미래 결과를 표현하지만 여러 값의 흐름, Backpressure와 Operator 조합을 제공하지 않는다. Reactive Pipeline은 Subscription 전까지 실행되지 않는 Lazy Publisher이며, Subscriber가 처리할 수 있는 양을 요청한다.

```text
Publisher -- onNext* --> Subscriber
          -- onComplete -->
          -- onError ---->

Subscriber -- request(n) --> Publisher
```

`onError`와 `onComplete`는 둘 중 하나만 오는 Terminal Signal이다. Terminal 이후 `onNext`를 보내면 안 된다.

## 실무 예제: 예약 조회 Pipeline

Reactive Driver를 사용하는 Port만 연결한다. 중간에 Blocking JPA Repository를 호출하면 Event Loop가 멈춘다.

```java
public interface ReactiveBookingPort {

    Mono<Booking> findById(long bookingId);

    Flux<Booking> findAllByPassengerId(long passengerId);
}
```

```java
@Service
@RequiredArgsConstructor
public class ReactiveBookingQueryService {

    private final ReactiveBookingPort bookingPort;
    private final ReactivePassengerPort passengerPort;

    public Mono<BookingDetailResource> getDetail(long bookingId) {
        Mono<Booking> booking = bookingPort.findById(bookingId)
                                           .switchIfEmpty(Mono.error(new BookingNotFoundException(bookingId)));

        return booking.flatMap(foundBooking ->
            passengerPort.findById(foundBooking.getPassengerId())
                         .switchIfEmpty(Mono.error(
                             new PassengerNotFoundException(foundBooking.getPassengerId())
                         ))
                         .map(passenger -> BookingDetailResource.from(foundBooking, passenger))
        );
    }
}
```

`map`은 동기 `T -> R`, `flatMap`은 비동기 `T -> Mono<R>` 변환에 사용한다. `Mono<Mono<R>>`를 만들지 않는다.

## 여러 조회의 동시성 제한

`flatMap`은 여러 Inner Publisher를 동시에 실행할 수 있다. 외부 API나 Connection Pool보다 큰 동시성을 열지 않도록 제한한다.

```java
public Flux<PassengerProfile> loadProfiles(List<Long> passengerIds) {
    int concurrency = 16;
    int prefetch = 32;

    return Flux.fromIterable(passengerIds)
               .distinct()
               .flatMap(
                   passengerClient::getProfile,
                   concurrency,
                   prefetch
               )
               .timeout(Duration.ofSeconds(2));
}
```

결과 순서가 입력 순서와 같아야 하면 `flatMapSequential`을 검토한다. `concatMap`은 한 번에 하나씩 처리하므로 순서는 보장하지만 Throughput이 낮다.

## Error 처리

```java
public Mono<FareQuote> quote(FareQuoteCommand command) {
    return pricingClient.quote(command)
                        .timeout(Duration.ofMillis(800))
                        .retryWhen(Retry.backoff(2, Duration.ofMillis(100))
                                        .filter(this::isTransientFailure)
                                        .onRetryExhaustedThrow((spec, signal) -> signal.failure()))
                        .onErrorMap(
                            TimeoutException.class,
                            cause -> new PricingTimeoutException(command.bookingId(), cause)
                        )
                        .doOnEach(signal -> observationRecorder.record(command.bookingId(), signal));
}
```

Retry는 멱등한 호출에만 적용하고 최대 횟수·Backoff·Jitter를 둔다. `onErrorResume(error -> Mono.empty())`로 실패를 숨기면 정상적인 부재와 장애를 구분할 수 없다.

## Blocking API 격리

교체할 수 없는 Blocking Library는 Event Loop가 아닌 제한된 Scheduler로 격리한다.

```java
public Mono<LegacyReport> generateReport(ReportCommand command) {
    return Mono.fromCallable(() -> legacyReportClient.generate(command))
               .subscribeOn(Schedulers.boundedElastic())
               .timeout(Duration.ofSeconds(10));
}
```

`publishOn`은 이후 Operator의 실행 Scheduler를 바꾸고 `subscribeOn`은 Source Subscription 위치에 영향을 준다. `boundedElastic`로 옮겼다고 Blocking 작업이 무한 확장 가능한 것은 아니다. Queue와 Timeout을 관측한다.

## Backpressure와 Buffer

생산자가 초당 10,000개를 만들고 Consumer가 1,000개만 처리하면 Memory가 계속 증가한다. 업무 의미에 따라 Buffer, Drop, Latest 또는 Producer 속도 제한을 선택한다.

```java
Flux<LocationUpdate> updates = locationSource.receive()
    .onBackpressureLatest()
    .groupBy(LocationUpdate::vehicleId)
    .flatMap(vehicleUpdates -> vehicleUpdates.sample(Duration.ofMillis(200)))
    .publish()
    .refCount(1);
```

위치는 최신 값이 오래된 값보다 중요하므로 `latest`가 가능하다. 결제·Outbox Event는 Drop하면 안 되므로 Durable Queue와 Consumer Lag 관리가 필요하다.

## HTTP Streaming

```java
@RestController
@RequiredArgsConstructor
public class VehicleLocationController {

    private final SubscribeVehicleLocationUseCase useCase;

    @GetMapping(value = "/vehicles/{vehicleId}/locations", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<VehicleLocationResponse>> subscribe(@PathVariable long vehicleId) {
        return useCase.subscribe(vehicleId)
                      .map(VehicleLocationResponse::from)
                      .map(response -> ServerSentEvent.builder(response)
                                                      .event("vehicle-location")
                                                      .id(response.sequence().toString())
                                                      .build())
                      .timeout(Duration.ofMinutes(30))
                      .doOnCancel(() -> useCase.recordCancellation(vehicleId));
    }
}
```

Client 재연결, `Last-Event-ID`, Heartbeat, Proxy Timeout과 느린 Subscriber 정책까지 설계해야 운영 가능한 SSE가 된다.

## Context와 관측성

ThreadLocal은 Signal이 Thread를 이동할 때 유실될 수 있다. Tenant ID와 Correlation ID는 Reactor Context로 전달하고 Framework의 Context Propagation을 사용한다.

```java
return bookingService.getDetail(bookingId)
                     .contextWrite(context -> context
                         .put(TenantContext.KEY, tenantId)
                         .put(CorrelationContext.KEY, correlationId));
```

Pipeline 전체를 `block()`으로 끝내거나 내부 Service에서 `subscribe()`를 직접 호출하지 않는다. Subscription과 Response Lifecycle은 Web Adapter 또는 Framework가 소유하게 한다.

## Test

```java
StepVerifier.create(service.getDetail(bookingId))
            .assertNext(resource -> assertThat(resource.bookingId()).isEqualTo(bookingId))
            .verifyComplete();

StepVerifier.withVirtualTime(() -> service.quote(command))
            .thenAwait(Duration.ofSeconds(3))
            .expectError(PricingTimeoutException.class)
            .verify();
```

시간 기반 Operator는 실제 Sleep 대신 Virtual Time으로 Test한다.

## 기억할 점

Reactive Code의 목적은 모든 Method를 `Mono`로 감싸는 것이 아니다. End-to-end Non-blocking Driver, 제한된 동시성, Backpressure, 명시적인 Error와 Context 전파가 함께 있어야 한다. Blocking 중심 Application이라면 Virtual Thread가 더 단순한 선택일 수 있다.
