# NVIDIA GPU에서 PyTorch로 Mini GPT 학습하기

앞선 글의 Embedding, Causal Self-Attention, Transformer Block과 Cross Entropy를 하나의 실행 가능한 Decoder-only Language Model로 연결한다. 완성 코드는 [mini_gpt.py](./Resources/mini_gpt.py)에 있다.

이 예제의 목적은 성능이 좋은 언어 모델을 만드는 것이 아니라 Tensor가 GPU에서 어떻게 Forward, Backward와 Optimizer Step을 거치는지 확인하는 것이다.

---

## 1. NVIDIA GPU 환경 확인

Host에서 Driver와 GPU를 확인한다.

```bash
nvidia-smi
```

PyTorch는 공식 설치 페이지에서 운영체제와 CUDA 조합에 맞는 Package 명령을 선택한다. 설치 후 다음을 확인한다.

```python
import torch

print(torch.__version__)
print(torch.version.cuda)
print(torch.cuda.is_available())
print(torch.cuda.get_device_name(0))
```

`nvidia-smi`의 CUDA Version은 Driver가 지원하는 최대 Runtime 계열이고, `torch.version.cuda`는 PyTorch Binary가 Build된 CUDA Toolkit Version이다. 두 값이 반드시 같아야 하는 것은 아니지만 Driver가 Runtime을 지원해야 한다.

---

## 2. CPU와 GPU는 서로 다른 Device Memory를 사용한다

```python
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model = model.to(device)
inputs = inputs.to(device)
targets = targets.to(device)
```

Model과 입력이 다른 Device에 있으면 연산할 수 없다. DataLoader에서 Page-locked Memory를 사용하고 비동기 복사를 요청하면 Transfer와 계산을 겹칠 여지가 생긴다.

```python
loader = DataLoader(dataset, pin_memory=True, num_workers=4)
inputs = inputs.to("cuda", non_blocking=True)
```

작은 Tensor를 자주 복사하면 GPU 계산보다 PCIe Transfer와 Kernel Launch가 더 비쌀 수 있다. Batch를 충분히 크게 만들고 전처리 병목을 Profile해야 한다.

---

## 3. Mini GPT의 Data 흐름

예제는 이해를 위해 Character Tokenizer를 사용한다.

```text
text
 -> character vocabulary
 -> token ids [B,T]
 -> token + position embedding [B,T,C]
 -> N transformer blocks [B,T,C]
 -> vocabulary logits [B,T,V]
 -> next-character cross entropy
```

Character Tokenizer는 Vocabulary가 작고 구현이 단순하지만 Sequence가 길어진다. 실제 LLM은 BPE 계열 Subword Tokenizer와 대규모 Corpus, Data Shuffling·Deduplication·Packing을 사용한다.

---

## 4. Causal Self-Attention 구현

전체 Q·K·V를 하나의 Linear Layer에서 계산한다.

```python
self.query_key_value = nn.Linear(embedding_dim, 3 * embedding_dim)
query, key, value = self.query_key_value(hidden).chunk(3, dim=-1)
```

그 후 `[B,T,C]`를 `[B,H,T,D]`로 나누고 PyTorch의 최적화된 Attention 연산을 호출한다.

```python
attended = functional.scaled_dot_product_attention(
    query,
    key,
    value,
    dropout_p=self.dropout if self.training else 0.0,
    is_causal=True,
)
```

`is_causal=True`가 미래 Token을 가린다. Training과 Evaluation에서 Dropout 적용 여부가 달라야 하므로 `self.training`을 확인한다.

---

## 5. Language Model Head와 Loss

마지막 Hidden State를 Vocabulary Logit으로 바꾼다.

```python
logits = self.language_model_head(hidden)  # [B,T,V]
loss = functional.cross_entropy(
    logits.reshape(-1, logits.size(-1)),
    targets.reshape(-1),
)
```

예제는 입력 Embedding과 출력 Head의 Weight를 공유한다. Vocabulary와 Embedding Dimension이 커질수록 이 Matrix가 많은 Parameter를 차지하므로 Weight Tying의 절감 효과도 커진다.

---

## 6. Mixed Precision

Tensor Core를 활용하고 Activation Memory를 줄이기 위해 CUDA에서 FP16 Autocast를 사용한다.

```python
use_amp = device.type == "cuda"
scaler = torch.amp.GradScaler("cuda", enabled=use_amp)

with torch.autocast(
    device_type=device.type,
    dtype=torch.float16,
    enabled=use_amp,
):
    _, loss = model(inputs, targets)

scaler.scale(loss).backward()
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
scaler.step(optimizer)
scaler.update()
```

Autocast는 모든 연산을 무조건 FP16으로 바꾸지 않고 안정성이 필요한 연산을 적절한 Dtype으로 실행한다. BF16을 지원하는 GPU에서는 넓은 지수 범위 때문에 Gradient Scaling 없이 더 안정적으로 학습할 수 있는 경우가 많다.

## 7. 실행

Repository Root에서 실행한다.

```bash
python3 'AI/LLM/Resources/mini_gpt.py' \
  --steps 500 \
  --batch-size 64 \
  --learning-rate 0.0003
```

출력에서 `device=cuda`인지 먼저 확인한다. Loss가 초기 Random 예측의 `log(V)` 부근에서 시작해 감소하는지 본다. 이 예제의 Data는 작고 반복적이므로 생성 품질이 아니라 학습 Pipeline 확인에만 사용한다.

---

## 8. GPU 실행은 비동기다

CPU는 CUDA Kernel을 Queue에 넣고 먼저 돌아올 수 있다. 시간을 잴 때 Synchronization이 없으면 실제 계산보다 짧게 측정한다.

```python
start = torch.cuda.Event(enable_timing=True)
end = torch.cuda.Event(enable_timing=True)

start.record()
logits, loss = model(inputs, targets)
end.record()
torch.cuda.synchronize()
print(start.elapsed_time(end), "ms")
```

`time.time()`을 쓴다면 측정 전후 `torch.cuda.synchronize()`가 필요하다. 다만 Synchronization을 Training Loop에 불필요하게 넣으면 CPU와 GPU의 Overlap을 막는다.

---

## 9. GPU Memory 관측

```python
print(torch.cuda.memory_allocated() / 1024**2, "MiB allocated")
print(torch.cuda.max_memory_allocated() / 1024**2, "MiB peak")
```

PyTorch Allocator는 재사용을 위해 Memory를 Reserve하므로 `nvidia-smi` 값과 `memory_allocated()`가 다를 수 있다. Out of Memory가 발생하면 다음 순서로 본다.

1. Batch Size와 Context Length를 줄인다.
2. Gradient Accumulation으로 Effective Batch를 유지한다.
3. Mixed Precision과 Activation Checkpointing을 적용한다.
4. Attention 구현이 `[T,T]` Matrix를 Materialize하는지 확인한다.
5. 사용하지 않는 Tensor Reference가 Graph를 붙잡고 있지 않은지 확인한다.

`torch.cuda.empty_cache()`는 살아 있는 Tensor를 해제하지 않으며 모델 자체의 필요 Memory를 줄이지 않는다.

---

## 10. Throughput과 Model FLOPs Utilization

GPU 사용률이 높다는 사실만으로 효율적인 Training은 아니다. 다음 지표를 함께 본다.

- Tokens per Second
- Step Time과 Data Loading Time
- Peak Allocated Memory
- Tensor Core 사용 여부
- Kernel 간 빈 구간과 CPU–GPU Synchronization
- 유효 Token 대비 Padding 비율

PyTorch Profiler와 Nsight Systems로 CPU, CUDA Kernel과 Memory Transfer Timeline을 분석할 수 있다. 작은 모델은 GPU를 채우지 못해 CPU가 더 빠를 수도 있다.

---

## 11. 여러 GPU로 확장하기

한 GPU를 넘으면 일반적으로 `DistributedDataParallel`로 Process당 GPU 하나를 사용한다. Model이 한 GPU Memory에 들어가지 않으면 Parameter, Gradient와 Optimizer State를 분할하는 FSDP나 ZeRO 계열 방식이 필요하다.

```text
Data Parallel: Model 복제, Batch 분할
Tensor Parallel: Layer의 행렬 연산 분할
Pipeline Parallel: Layer 묶음을 Stage로 분할
```

통신 비용 때문에 GPU 수를 두 배로 늘려도 처리량이 정확히 두 배가 되지는 않는다. NCCL 통신, Network Topology와 Global Batch 변화가 수렴에 미치는 영향까지 측정한다.

---

## 12. 이 예제와 실제 LLM Training의 차이

실제 Training에는 대규모 Tokenizer와 Dataset, Learning Rate Warmup·Decay, Validation, Checkpoint, Distributed Training, Fault Tolerance와 Evaluation이 필요하다. 또한 Data License, 개인정보와 유해 Content 처리도 모델 구조만큼 중요하다.

Mini GPT는 원리를 투명하게 보여주기 위해 기능을 줄였다. 이 Code를 그대로 Production 학습 Framework로 확장하기보다 각 개념을 이해한 뒤 검증된 Distributed Training Stack을 선택하는 편이 안전하다.

---

## 마무리

GPU는 Transformer Algorithm을 바꾸지 않는다. 같은 Tensor 연산을 많은 Core에서 병렬로 수행한다. 성능을 얻으려면 연산량뿐 아니라 Shape, Dtype, Memory 이동과 Synchronization을 함께 이해해야 한다.
